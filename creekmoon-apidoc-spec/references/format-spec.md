# 接口文档格式规范参考

本文档为 `creekmoon-apidoc-spec` 技能的详细格式参考，定义标准化 API 文档的结构与约定。

## 文档层级与编号

- **一级标题（#）**：平台/系统名称 + 资源接口说明
- **二级标题（##）**：主章节，按 1、2、3... 编号
- **三级标题（###）**：子章节，按 1.1、1.2、2.1、2.2... 编号
- **四级标题（####）**：接口内子节，如 6.1、6.2、6.3

## 单接口章节结构

每个业务接口按以下结构组织：

```
## N. [业务模块] 接口

### N.1 接口说明
- 接口地址
- 接口功能
- 请求方式
- Content-Type

### N.2 请求参数
- 参数表格（5列）
- 请求示例 JSON

### N.3 响应参数
- 响应字段表格（5列）
- 响应示例 JSON

### N.4 [可选] 子结构说明
- 当响应中有嵌套对象时，单独列出字段说明
```

## 章节复用规则与示例

### 默认策略

每个接口的请求参数和响应参数**必须完整列出字段表与示例 JSON**。
读者在当前章节就能理解本接口，不需要跳到任何其他章节。

### 允许复用的条件

仅当以下三个条件**同时成立**时才允许简写：

1. 结构与被引用接口**完全一致**（包括字段名、类型、说明）；
2. 该结构字段较多，全量重复会大幅拉长篇幅；
3. 使用**可点击的 Markdown 链接**，不得用纯文字描述。

### 复用目标选择原则

- **优先做法**：将共享结构提炼到 `## 5. 通用数据结构` 或 `## 附录` 中，以命名的 `###` 标题定义（如 `### 报价项结构`），然后在接口章节中用链接指向该结构。
- **禁止做法**：直接链接到另一个接口的 `响应参数` 章节（如 `见 #### 7.1.2 响应参数`）。

### 使用引用时的必要补充

即使引用了共享结构，也必须在同一章节说明：

- 本接口的差异点（字段增减、含义变化）；
- 如果确无差异，注明"完全一致，无额外字段"；
- 补充本接口特有的状态说明或业务行为。

## 表格列定义

### 请求参数表

| 列名   | 含义                         | 取值示例                          |
|--------|------------------------------|-----------------------------------|
| 字段   | 参数名                       | keyword, pageNum                  |
| 类型   | 数据类型                     | String, Integer, Long, Boolean     |
| 是否必填 | 是否必传                   | 是 / 否 / -                      |
| 说明   | 参数含义；字典字段需明确写“字典字段”并给出对接方可见的取值说明（不暴露内部字典类型编码）；非字典但常见值字段需直接提示常见值 | 关键词，支持模糊查询 |
| 示例值 | 调用时示例                   | "示例值", 1                       |

### 响应参数表

| 列名   | 含义                         | 取值示例                          |
|--------|------------------------------|-----------------------------------|
| 字段   | 返回字段名                   | id, name                          |
| 类型   | 数据类型                     | String, Integer, Long, Object     |
| 是否必传 | 是否一定返回               | 是 / 否 / -                      |
| 说明   | 字段含义；字典字段需明确写“字典字段”并给出对接方可见的取值说明（不暴露内部字典类型编码）；非字典但常见值字段需直接提示常见值 | 唯一标识，用于关联查询 |
| 示例值 | 返回示例                     | "xxx-id-xxx"                      |

## 字段说明分类规则

- **普通字段**：保持常规写法，说明中写清业务含义即可
- **字典字段**：说明中必须明确写出「字典字段」，并在说明或附录中列出对接方可见的「字典值 → 含义」对照；**不要暴露内部字典类型编码**（即系统内部按哪个 code 大类存储的标识，对接方无需关心）
- **非字典但常见值字段**：说明中直接写明其为**非字典字段**，并补充常见值；示例值列应体现这些常见值，不要求写入附录

## 字段说明写法示例

- **普通字段**：`客户名称`
- **字典字段**：`字典字段，取值见 [附录「订单状态」](#订单状态)`
- **非字典但常见值字段**：`非字典字段，常见值如 wechat、alipay、bank`

锚点写法说明：Markdown 标题自动生成锚点，使用标题文字即可（中文直接保留，标点去掉，空格替换为 `-`）。例如附录小节 `### 订单状态`，对应链接为 `[附录「订单状态」](#订单状态)`。

> 反例（不要这样写）：`字典 lastMileServiceLevel：OD-室外配送、IDD-室内配送`——`lastMileServiceLevel` 是内部字典类型编码，不应出现在对外文档中。

## 统一响应 data 结构（分页接口）

```json
{
  "total": 100,
  "list": [...],
  "pageNum": 1,
  "pageSize": 10,
  "pages": 10,
  "size": 10,
  "startRow": 1,
  "endRow": 10,
  "prePage": 0,
  "nextPage": 2,
  "isFirstPage": true,
  "isLastPage": false,
  "hasPreviousPage": false,
  "hasNextPage": true
}
```

核心必填字段：`list`、`total`、`pageNum`、`pageSize`、`pages`。其余可根据实际实现保留或省略。

## 附录枚举表格式

字典字段和固定枚举字段都应在附录中提供取值对照，附录只列「字典值 → 含义」，不出现内部字典类型编码；非字典但常见值字段只在参数表说明和示例值中体现，不进入附录。

每个枚举/字典对照小节使用独立的 `###` 标题，标题即为 Markdown 锚点目标，供参数表说明列的链接跳转使用。

```
### 10.1 通用枚举值对照

#### 订单状态
| 枚举值  | 含义   |
|---------|--------|
| type_a  | 类型A  |
| type_b  | 类型B  |

#### 配送服务等级
| 枚举值 | 含义   |
|--------|--------|
| OD     | 室外配送 |
| IDD    | 室内配送（默认）|
| 2      | 已结束 |
```
