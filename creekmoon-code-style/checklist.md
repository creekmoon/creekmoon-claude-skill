# Creekmoon Code Style — 审查检查清单

逐项核对，每项标注关联规则编号和违规信号（快速判断特征）。

---

## 方法结构与组织

- [ ] **方法体是否按 0-4 分区组织？** [R9]
  - 违规信号: 校验散落在方法各处；默认值赋值和业务执行交替出现；没有清晰的阶段边界

- [ ] **空行是否仅用于分区切割？** [R9]
  - 违规信号: 同一分区内出现多次空行；块注释与紧跟的代码之间有空行；空分区保留了占位空行

- [ ] **各阶段顶部是否使用了有意义的块注释？** [R9]
  - 违规信号: 使用 `//` 做阶段标题；使用无价值占位注释如 `// 1. 参数声明区域`；缺少阶段注释
  - 正确格式: 单行 `/* xxx */`，多行 `/* ... */`

- [ ] **阶段内部是否避免了与代码行为重复的注释？** [R9]
  - 违规信号: `/* 校验订单不为空 */ if (order == null)`；`/* 打日志 */ log.info(...)`

- [ ] **是否遵循 Happy Path First？** [R5]
  - 违规信号: 主逻辑嵌套在 `if (xxx != null)` 内部；先做空声明再在分支里赋值

- [ ] **入参变量是否合理复用？** [R9]
  - 违规信号: 额外声明了与入参同含义的新变量（如 `String bizCode = code`）

---

## 方法设计与命名

- [ ] **方法是否保持内聚？** [R1]
  - 违规信号: 方法体内存在不必要的多层跳转或间接层

- [ ] **是否存在单调用点 private 方法的过度抽取？** [R2]
  - 违规信号: private 方法只有一个调用点；抽取后需要频繁跳转阅读上下文

- [ ] **方法名是否准确覆盖了实现行为（Name == Behavior）？** [R3]
  - 违规信号: 方法内部存在方法名未提及的**业务独立**副作用（通知、级联删除、写入其他业务表、发消息）
  - 豁免: 内生耦合的附带行为（如更新 updateTime、填充审计字段）不需要体现在方法名中
  - 违规信号: 方法内部有前置检查并可能静默 return，但方法名未体现（缺少 `IfExists` 等）

- [ ] **若方法名包含 `AndXxx`，是否保证无论什么情况都会执行 `Xxx`？** [R6]
  - 违规信号: `saveUserAndNotice` 内用 boolean 参数控制是否通知

- [ ] **若方法名以 `try` 开头，是否符合 tryXxx 约定？** [R8]
  - 违规信号: tryXxx 方法向外抛出异常；返回类型不是 void/boolean；可能打断外部事务
  - 违规信号: 缺少 try-catch；日志未标注"可能是正常情况无须理会"

---

## 入参与参数风格

- [ ] **Service public 方法入参是否优先使用基本类型/简单对象？** [R4]
  - 违规信号: public 方法入参使用聚合对象（如 Order、UserInfoDTO），而对象可通过简单查询获得
  - 例外确认: 是否属于 private 方法？是否直接为 Controller 层服务？对象获取代价是否确实很高？

---

## 防御性编程与可读性

- [ ] **主流程强依赖的关键校验是否已前移到 0/3 区域？** [R9][R10]
  - 违规信号: 业务执行阶段（区域 4）出现大量 `if (xxx == null)` 防御性分支

- [ ] **是否避免把空判断+取值+兜底堆进同一个参数表达式？** [R10]
  - 违规信号: `StrUtil.nullToEmpty(obj != null ? obj.getXxx() : null)` 这类嵌套表达式出现在调用参数中

- [ ] **是否避免把"声明+兜底"揉成一次性三元表达式？** [R9][R10]
  - 违规信号: `String x = condition ? sourceA.getX() : sourceB.getX()` 而本应分阶段先声明再兜底

- [ ] **是否存在以"性能"为由写出的裸魔数/位运算技巧？** [R11]
  - 违规信号: 手写字节魔数做文件类型检测；手动内联标准库能完成的操作

---

## 流式转换与后续逻辑

- [ ] **纯 CPU 转换是否使用 Stream 并顶部写了步骤注释？** [R7]
  - 违规信号: 纯转换逻辑存在多个仅使用一次的中间变量；缺少顶部步骤注释
  - 违规信号: I/O 操作混在 stream 链路中

- [ ] **可选后续逻辑是否通过回调组合或显式命名处理？** [R6]
  - 违规信号: 方法名未提及的后续操作硬编码在方法内部
