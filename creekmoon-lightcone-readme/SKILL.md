# 项目 README 架构化写作技能

这个技能只做一件事：**把仓库理解结果整理成一份适合放在根目录的 README。**

目标不是写“代码说明书”，而是让第一次接触项目的人尽快回答下面几个问题：

- 这是个什么系统，解决什么问题
- 业务主链路怎么跑
- 数据以什么对象在各模块间流转
- 核心生命周期分几段
- 模块之间怎么协作
- 新同学应该先从哪里进入

如果仓库已经有 `README.md`，默认做 **增强式重构**，保留有价值内容，不直接推倒重写。

## 适用场景

- 用户要新建项目 README
- 用户要重构、补强、精简现有 README
- 用户想补充“业务流程 / 数据流 / 生命周期 / 模块边界 / 新人上手”
- 用户说“这个 README 太像工程说明，不像项目说明”
- 用户希望用更通用的方式沉淀项目背景文档

## 不适用场景

- 只写接口字段、请求响应示例的 API 文档
- 只写部署步骤、运维参数、环境变量说明
- 只做某一个模块的详细设计文档
- 需要逐行解释代码实现细节

这些内容可以存在于仓库里，但不应该占据根目录 README 的主轴。

## 第一原则

- README 先回答“项目如何运转”，再回答“代码怎么组织”
- 优先讲 **业务链路**，其次讲 **架构装配**
- 只写架构层与模块边界，不写类、方法、SQL、字段校验细节
- 信息要能支撑新人快速参与开发，而不是堆满知识点
- 图和表都要短，默认只保留最关键的一层
- 结论强度不能超过证据强度
- 先写“代码和现有文档已经证明了什么”，再写“基于证据可做的保守判断”
- 目录名、类名、DTO 字段、注释、行业术语只能作为线索，不能单独证明项目定位、系统边界或完整业务域
- 判断不清时，优先使用更中性的定位句；必要时只补 1-2 行待确认说明，不把 README 写成审计报告

## 开始前先判断文档模式

先判断项目更适合哪种 README 主轴：

### 1. 业务系统型

适用于工单、交易、履约、内容处理、CRM、运营平台等有明显业务闭环的系统。

主轴优先级：

1. 业务背景与主线
2. 核心业务执行流程
3. 数据流转形式
4. 核心业务生命周期
5. 模块间对接关系
6. 工程结构补充

### 2. 能力平台 / 基础设施型

适用于框架、SDK、平台能力层、中间件、工具型项目。

主轴改成：

1. 项目定位与能力边界
2. 典型使用链路
3. 输入 / 输出 / 状态变化
4. 模块职责与扩展点
5. 运行时架构与接入方式

### 3. 混合型 / 暂未定型

适用于既有明显业务流程，又承担平台能力、集成适配、通用框架职责的仓库；或者当前证据还不足以把它稳定归到前两类之一。

主轴建议：

1. 项目当前承担的主要职责
2. 核心输入 / 输出 / 关键对象
3. 典型处理链路
4. 模块协作与边界
5. 工程结构补充

判断规则：

- 先看入口、主流程、模块协作和已有项目文档，再看目录名和命名
- 如果去掉命名暗示后，项目定位仍然成立，才使用较强定性
- 如果判断不清，先保守归类为“混合型 / 暂未定型”，不要为了套模板硬把仓库写成某类完整产品
- 当 README 主轴依赖关键边界判断、但证据不够时，先补问 1 个问题；如果用户暂时无法补充，就采用更中性的表述

## 信息收集与探索优先级

先按优先级收集能决定 README 主轴的信息，不要一上来就沉进实现细节：

| 优先级 | 目标 | 读什么 |
|--------|------|--------|
| 1 | 项目已有定位说明 | 现有 `README.md`、`CLAUDE.md`、架构文档、项目背景文档、根构建文件 |
| 2 | 系统边界与运行形态 | 启动入口、主程序入口、运行配置、模块装配方式 |
| 3 | 本仓库直接拥有的能力 | Controller、Job、RPC、Message Handler、核心模块协作 |
| 4 | 外部对接范围 | `client`、`adapter`、`partner`、`api`、`integration` 等目录与调用点 |
| 5 | 主链路与关键对象 | 核心领域目录、状态流转代码、关键输入输出载体 |
| 6 | 新人入口 | 启动脚本、onboarding 文档、最常用的开发入口 |

补充规则：

- 命名只能作为辅助线索，不能替代项目文档、入口、主流程和模块协作证据
- 如果项目很大，先找“贯穿全链路的入口”而不是“最底层的工具类”
- 如果读到第 3-4 优先级后仍无法确定主轴，先按“混合型 / 暂未定型”处理，而不是继续脑补

## 写正文前必须先完成候选结论审查

在信息收集完毕后、第一段正文落笔前，**必须**先完成下面这些结论的审查，再开始写正文：

- 项目定位
- 业务主线或典型使用链路
- 核心对象
- 核心模块角色
- 对外系统关系

先整理一个轻量表：

| 候选结论 | 证据类型 | 当前判断 | README 写法 |
|---------|----------|----------|-------------|
| `{结论}` | 直接证据 / 弱线索 | 可直接写 / 需保守改写 / 暂不写入 | 主结论 / 保守表述 / 待确认说明 |

证据判断规则：

- **直接证据**：现有 README、架构文档、启动入口、主流程调用链、核心模块协作、状态流转、核心持久化、运行配置
- **弱线索**：目录名、包名、类名、DTO 字段、注释、第三方协议名、单个 client / adapter / controller

审查规则：

1. **事实审查**
   - 如果去掉命名暗示后，这个结论仍然成立，才可以保留较强表述
   - 只有弱线索时，不能直接写“本系统就是某类完整业务系统”或“该模块负责完整业务域”

2. **边界审查**
   - 我写的是“本仓库直接拥有的能力”，还是“本仓库对接 / 适配 / 消费的外部能力”？
   - 我是不是把外部系统术语、回调协议、DTO 命名误写成了内部业务边界？

3. **反向否证**
   - 如果同时存在激进解释和保守解释，优先选择保守解释
   - 证据不足但又影响读者理解时，只补 1-2 行待确认说明；如果不影响主线，宁可不写

推荐改写方式：

- **可直接写**：证据充分，且能回到入口、主流程、模块关系或现有项目文档
- **保守改写**：写成“按当前代码结构看更像…”、“当前更接近…”、“从入口和模块协作看主要承担…”
- **暂不写入**：只有命名、注释、局部接口、第三方术语，没有更强证据支撑

## 输出结构

默认按下面顺序组织 README。除非项目类型明显不适配，否则不要轻易跳过前 6 节。

混合型 / 平台型项目中，第 4-6 节可按项目实际情况省略，或替换为“典型使用链路 / 输入输出 / 能力生命周期”。

### 1. 标题与一句话定位

必须包含：

- 项目名称
- 1 句话说明它解决什么问题
- 极短技术栈摘要（可选）

### 2. 一屏摘要

这一节是整个 README 的核心入口，必须做到“快速扫完就有心智模型”。

建议包含：

- **一句话方案**：`入口 -> 核心处理 -> 执行/回传 -> 结果消费`
- **1 张核心流程图**
- **1 张简短事实表**

事实表推荐列：

| 项 | 说明 |
|----|------|
| 系统形态 | 单体 / 微服务 / 多模块 / SDK |
| 核心主线 | 业务主链路的一行总结 |
| 主要入口 | HTTP / Job / RPC / Message / CLI |
| 主要对象 | 请求 / 任务 / 记录 / 资源 / 事件 / 结果等 |

### 3. 业务背景与主线

这一节解释“为什么有这个系统”，但不要变成 PRD 复述。

只回答：

- 它更像面向业务闭环的系统、面向能力复用的平台，还是两者混合
- 业务主线由哪几段组成
- 哪些模块是主角，哪些是辅助

如果项目定位仍然模糊，优先描述“它处理什么输入、产出什么结果、连接哪些模块”，不要为了显得完整而强行命名系统类型。

### 4. 核心业务执行流程

这一节说明“业务是怎么跑起来的”。

必须包含：

- 一张简洁流程图
- 一张阶段表：阶段 / 负责模块 / 关键产出

流程图规则：

- 默认 `5-7` 个节点
- 只画主链路
- 省略登录、参数校验、DTO 转换等常识步骤
- 节点命名尽量用业务动作，不要用技术组件名

推荐示例：

```mermaid
flowchart LR
  A[触发来源] --> B[接收处理]
  B --> C[核心逻辑]
  C --> D[结果产出]
  D --> E[消费方]
```

### 5. 数据流转形式

这一节重点写 **业务对象** 如何流转，不要只写 `Service -> DB -> Redis`。

必须包含：

- 一张业务对象流转图
- 一张对象表：对象 / 产生位置 / 消费位置 / 作用

优先使用贴近仓库真实语义、又不过度脑补的对象名：

- 请求
- 任务
- 指令
- 记录
- 资源
- 事件
- 结果
- 状态快照

如果仓库里已经有明确、稳定、可证实的领域对象名，可以直接沿用；不要为了让 README 更“像业务文档”而自行补造术语。

如果只是技术资源流转，可作为补充视角放到后面，而不是替代业务对象流转。

### 6. 核心生命周期

这一节回答“这个系统的核心对象如何从进入到收敛”。

推荐列：

| 生命周期阶段 | 状态含义 | 关注模块 |
|--------------|----------|----------|

常见阶段可按需选用：

- 进入
- 识别 / 标准化
- 决策
- 执行
- 跟踪
- 消费
- 收敛

如果项目不是业务系统，可以把“生命周期”替换成“能力调用生命周期”或“任务执行生命周期”。

### 7. 核心模块对接关系

这节用来讲清楚“谁和谁对接、以什么方式对接、为什么对接”。

推荐列：

| 模块 | 上游输入 | 下游输出 | 对接方式 | 架构角色 |
|------|----------|----------|----------|----------|

注意：

- 这是“协作关系表”，不是“包目录清单”
- 重点写输入、输出和角色，不展开字段
- 如果一个模块只是公共支撑，不要写得像主链路节点

### 8. 易混淆模块对比

只有当模块边界容易混淆时才补这一节。

推荐列：

| 模块 | 更像什么 | 核心职责 | 不重点负责 |
|------|----------|----------|------------|

这个表很适合解决下面这些问题：

- 两个模块名字接近，但职责不同
- 一组模块都和“任务 / 流程 / 资源”有关，容易混
- 仓库规模大，新人不知道先看哪个模块

### 9. 工程视角补充

前面的业务视角讲完后，再补工程结构。常见内容包括：

- Maven / npm / workspace 模块职责
- 运行时架构
- 入口类型对比
- 包域划分
- 新同学上手

如果现有 README 已经有这些内容，优先保留并微调，不要整段删掉。

## 更新已有 README 的规则

如果仓库已经有 `README.md`，遵守下面的默认策略：

- **保留原有有价值内容**
- **优先前置新增“业务视角”章节**
- 如果旧章节比较窄，可以改名为：
  - `局部视角`
  - `技术资源视角补充`
  - `工程装配视角`
- 启动方式、接口文档、运行入口等通常保留
- 只在明显重复、冲突或失效时做删改

不要因为想追求“统一风格”就把原文全部重写掉。

## 图表约束

### 图的约束

- 默认 `1-2` 张图足够
- 单张图优先不超过 `7` 个节点，硬上限 `10`
- 只表达主干链路，不表达所有分支
- 解释优先放在图下方 1-3 行，不把整段说明塞进图里

### 表的约束

- 一张表只解决一个问题
- 默认 `4-6` 行最佳
- 字段少而准，避免把表写成“压缩版设计文档”

## 语言与表达规则

- 先结论，后展开
- 段落短，句子直
- 尽量用业务词汇，不要满篇技术术语
- 不写类名、方法名、SQL、配置 key、注解细节
- 不重复同一个信息在图、表、段落里各讲一遍
- 对存在歧义的地方，要明确写“当前按 X 理解”
- 项目定位、模块角色、对象定义要能回到具体证据，不要只因为名字“像”就下结论
- 推断型表述要带保守措辞，例如“按当前代码结构看更像…”、“当前更接近…”
- 把“本仓库直接拥有的能力”和“本仓库对接 / 适配 / 消费的外部能力”分开写
- 第三方协议名、行业术语、外部系统字段可以作为背景线索，但不能自动补全成完整业务边界

## 常见反模式

- README 只剩模块列表，看不出业务主线
- 数据流只写成 `Service -> DB -> Redis -> 外部接口`
- 生命周期只覆盖一个很窄的子流程
- 图太大，把所有模块全画进去
- 表太密，读者 30 秒内抓不到结论
- 写成“详细设计”而不是“项目总览”
- 为了精简把新人真正需要的入口信息也删掉
- 只因为目录名、类名、DTO 字段，就把项目直接写成某类系统
- 把外部接口、SDK、回调协议、适配层痕迹写成本仓库自有核心能力
- 为了让 README 更完整，脑补出代码里并没有闭环证明的业务主线

## 输出前检查清单

完成 README 前，至少检查下面这些问题：

- [ ] 读者能在 1 分钟内知道项目是做什么的
- [ ] 有一条明确的业务主链路
- [ ] 有业务对象视角的数据流，而不只是技术资源流
- [ ] 有核心生命周期，而不只是某个局部流程
- [ ] 模块交互说明的是协作关系，不是目录清单
- [ ] 原有 README 的有效内容没有被粗暴删除
- [ ] 新同学读完后知道先看哪几个模块或入口
- [ ] 没有把目录名、模块名、类名直接上升成系统定位
- [ ] 没有把外部协议、第三方术语或对接代码误写成仓库自有能力
- [ ] 没有把局部 controller / client / DTO 的存在扩展成完整业务链路
- [ ] 对证据不足的地方，已经改成保守表述或省略，而不是继续强定性

### 整体视角收尾

- [ ] 这份 README 更像“项目总览”还是“目录说明”？
- [ ] 如果是第一次接触项目的人，他能否立刻知道项目背景、主链路和入口？
- [ ] 有没有把众所周知的步骤写得太重？
- [ ] 有没有哪一句是因为术语或命名很像，就被我写成了已经证实的事实？

如果答案不够明确，继续压缩细节，补强主线。

## 建议输出风格

如果用户没有指定格式，优先输出为：

- 根目录 `README.md`
- 标题简短
- 章节层级不超过两层
- Mermaid 图简洁
- 表格简洁
- 以“业务视角在前，工程视角在后”为默认顺序

## 示例触发语句

下面这些请求都应该优先触发本技能：

- “帮我重写一下项目的 README，让新人能看懂业务流程。”
