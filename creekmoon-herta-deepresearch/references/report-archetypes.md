# Report Archetypes

本文件是报告范式库，供 `report-composer` 判定意图、推导目录时参考。

**核心立场：目录从用户真实问题里长出来，不是从模板里抄出来。**

下面每种范式给的骨架都是「常见起点」，不是「必须照填的模板」。撰稿官要先认出这次到底是哪类诉求，再决定要不要、怎么改这套骨架。同一份研究，问法不同，报告就该长得不一样。

## 怎么判意图

先读原始诉求原话，问自己三件事：

1. **用户到底想拿这份报告干什么？** 是确认一个事实、做一个决策、看懂一套机制、还是评估一个风险？
2. **用户显式问了哪几个问题？** 把每个问号、每个"是否/能不能/为什么/怎么样"都拎出来，这些就是报告必须正面回答的点。
3. **用户读完最想得到什么？** 一个明确结论、一张全景图、一份改动清单、还是一个取舍判断？

判完意图，再去下面的范式库里找最接近的一种（或两种混合），把它的骨架改写成贴合本次问题的目录。

## 通用底线（所有范式共用）

无论哪种范式，最终报告都要满足：

- 第一屏直接回答用户最核心的那个问题，而不是先讲"这件事是什么"的背景
- 用户每个显式问题，正文里都能找到正面、明确的回答
- 有一条主线，所有小节服务这条主线
- 结论可回溯到证据锚点
- 图按需出现：能帮理解就画，纯靠文字/表格说得清就不硬画
- 不出现未确认/待验证栏目；证据不足的判断留在审查层

## 范式一：事实确认型

**典型诉求**：「我想确认一下，X 是不是 Y」「目前是不是用了 Z」「有没有用到某种特殊用法」。用户已经有假设，要的是被证实或被纠正。

**第一屏先给**：对每个待确认点的直接判定（是 / 不是 / 部分是，附一句关键澄清）。

**建议骨架（可改）**：
- 结论速览：逐条回答用户的确认点（一个判定 + 一句澄清）
- 逐点展开：每个确认点单独一节，讲清「事实是什么、为什么是这样、容易误判在哪」
- （按需）一张图：当多个事实点之间有数据流向或边界关系，画一张帮读者把它们串起来
- 证据回溯

**不要硬塞**：不要为了模板完整，给一个纯确认型问题套上"业务全景→生命周期→模块对接→为什么这样组织"。用户没问业务怎么演化，就别讲生命周期。

## 范式二：决策支持型

**典型诉求**：「能不能迁到 X」「该不该上 Y」「这样改可行吗」「有没有坑」。用户要做一个决定，需要一个可执行的判断 + 依据 + 代价。

**第一屏先给**：明确的可行性结论（可行 / 不可行 / 有条件可行），以及最关键的那个前提或代价。

**建议骨架（可改）**：
- 结论与前提：能不能做，卡点在哪，代价是什么
- 现状关键事实：支撑这个结论的几条决定性事实
- 可行性逐项分析：分维度判定（能力 / 兼容性 / 工作量 / 风险），用对比表最有效
- 改造范围与改动清单：哪些模块要动、改成什么
- 证据回溯

**不要硬塞**：决策型报告的重点是「结论 + 代价 + 改动清单」，不是业务叙事。不要把"为什么系统当年这样设计"写成主线，那只是背景，点到即止。

## 范式三：机制解释型

**典型诉求**：「X 到底是怎么运作的」「这套流程怎么跑通的」「为什么会出现这个结果」。用户要看懂一个机制的内在逻辑。

**第一屏先给**：这套机制的核心判断——真正决定结果的是什么。

**建议骨架（可改）**：
- 核心结论：这套机制本质在干什么、关键决定点在哪
- 主流程：从起点到终点怎么走，关键转折在哪（通常需要一张流程图/时序图）
- 关键判断点：哪几个判断真正改写结果，去掉会怎样
- 例外与回退：什么情况下不走主路
- 证据回溯

**不要硬塞**：不要把每个步骤都画进图、写进正文。机制解释的价值在「哪几个点决定结果」，不在「完整复述每一步」。

## 范式四：业务理解型

**典型诉求**：「帮我看懂这块业务」「新人接手要先理解什么」「这个系统整体是怎么回事」。用户要建立一套业务心智模型。

**第一屏先给**：这件事在业务上到底是什么，最该先记住的几条认知。

**建议骨架（可改，这是原版默认结构，仅适用于本范式）**：
- 先看懂这件事：本质一句话 + 最值得先记住的几条判断
- 一张图先看整体：业务全景图
- 数据和业务怎么流：主流程
- 核心业务生命周期：对象的关键阶段（有明显生命周期才写）
- 模块如何对接：职责分工（模块分工是主线才写）
- 为什么这样组织：业务诉求与取舍
- 少量例外与阅读提醒
- 证据回溯

**注意**：这套骨架只在用户真的要"理解业务全貌"时才适用。它是范式之一，不是所有报告的默认模板。把它套到事实确认或决策型问题上，就会产生 AI 味和答非所问。

## 范式五：风险评估型

**典型诉求**：「这里有什么风险」「上线前要担心什么」「哪里最容易出事」。用户要一张可管理的风险地图。

**第一屏先给**：最该优先处理的那个风险，以及整体风险量级判断。

**建议骨架（可改）**：
- 风险总览：按影响/可能性排序，最重的在前
- 逐项风险：每条讲清「触发条件、后果、当前把握、怎么减轻」
- （按需）一张图：风险点在系统流程里的位置
- 证据回溯

**不要硬塞**：风险报告不需要业务全景图凑数，也不需要"为什么这样组织"。重点是排序和可操作性。

## 范式六：对比选型型

**典型诉求**：「A 和 B 怎么选」「现状方案和新方案对比」「换成 X 比现在好在哪」。用户要在选项间做选择。

**第一屏先给**：推荐选哪个，以及一句话理由。

**建议骨架（可改）**：
- 推荐结论：选哪个，为什么
- 对比维度表：逐维度对比（这是对比型的核心，表格最有效）
- 关键差异展开：只展开真正影响选择的差异
- 迁移/落地代价：选了之后要付出什么
- 证据回溯

## 范式七：排查复盘型

**典型诉求**：「为什么会出这个问题」「这个 bug 根因是什么」「复盘一下这次故障」。用户要根因 + 改进。

**第一屏先给**：根因一句话 + 影响范围。

**建议骨架（可改）**：
- 根因结论：到底是什么导致的
- 触发链路：从诱因到现象怎么一步步发生的
- 影响范围：谁受影响、受什么影响
- 改进方向：怎么避免再次发生
- 证据回溯

## 混合与裁剪

真实诉求经常是混合的。你这次的例子就是 **事实确认型 + 决策支持型** 的混合：先确认"数据源是不是 Kafka、落库是不是 Mongo、有没有特殊语法"（事实确认），再判断"能不能无缝迁 MySQL"（决策支持）。

混合时的做法：
- 以用户最终目的为主范式（这例里是「能不能迁」，决策支持为主）
- 把另一类诉求作为前置章节（先把几个事实确认掉，再给迁移结论）
- 目录直接按「用户问题清单 + 决策结论 + 改动清单」组织，不要两套模板叠加

裁剪原则：
- 用户没问的维度，不要为了"显得完整"硬加章节
- 一个章节如果不能对应到某个用户问题或主线判断，删掉
- 章节顺序按「用户最想先知道的」排，不按「研究的工作顺序」排

## 一句话原则

**先认出用户到底要什么，再决定报告长什么样；范式库是参考骨架，不是必填模板。能用三节说清的事实确认，就不要硬撑成八节的业务叙事。**
