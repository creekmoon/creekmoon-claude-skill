# Style Guide

本文件约束最终报告的写法，不约束搜证过程。

## 目标

让报告读起来像“一个想清楚了的人，在用平常话直接回答你这次的问题”，而不是“把研究底稿按模板摊开”。

报告的核心任务：贴合用户这次的真实诉求，把每个问题正面答到。报告该长什么样由意图决定（见 `references/report-archetypes.md`），本文件约束的是不管哪种意图都要守的写法质量。

## 先记住这件事

- 好报告不是更长，而是更快让读者拿到他这次想要的答案
- 报告结构服务用户诉求，不服务模板惯例
- 图是帮读者理解的，不是凑数的；不需要图就不画
- 文字的任务是定方向和补关键理解，不是把图重说一遍
- 技术说明点到即止，永远不要成为报告主线

## 直白表达规则

- 优先使用平常说法，把一句话说到读者第一次读就懂
- 可以使用必要技术词，但技术词只在它真的更准确时才出现
- 不要用黑话、包装词、互联网公司话术撑句子
- 不要用大段比喻替代说明；能直说，就直说
- 如果一句话必须靠抽象词才成立，说明它还没有写清楚

### 常见替换

优先这样写：
- `从外面进来的消息`，少写 `扇入`
- `往下游发消息`，少写 `扇出`
- `整条链走完`，少写 `闭环`
- `触发点 / 时机`，少写 `钩子`
- `先放一个位置`，少写 `占位`
- `把一批任务拆开处理`，少写 `摊开`、`收口`
- `说明这件事为什么这样设计`，少写 `赋能`、`抓手`

## 三问原则

写完每个大节后，问自己：
1. **是什么**：这一节有没有说清这件事的业务形态？
2. **怎么样**：这一节有没有说清业务怎么流、对象怎么变、模块怎么接？
3. **为什么**：这一节有没有说明为什么会这样设计？

如果一个大节对这三问一个都回答不了，说明这节应该删掉或重写。

## 起手规则

这是整个写作规范里优先级最高的规则。

**每个大节先用 1 段短文字立住核心判断，再决定要不要补图、列表或表格。**

这段短文字通常只需要做到三件事：
1. 先说这一节要帮读者看懂什么
2. 再说关键变化或关键边界是什么
3. 必要时补一句“接下来这张图先看哪里”

不要求为了模板完整硬写 2-4 句铺垫。能 1 句讲清，就只写 1 句。

## 图后说明规则

图后说明默认从简。

每张图通常只需要补 1-2 句，优先回答：
1. 这张图最该先看哪里
2. 如果有必要，图里省略了什么

不要固定写“三连解释”。只有在省略内容可能让读者误解时，才补一句“为什么省略不会误导”。

合格示例：
> 先看中间的“库存匹配”节点，它决定后面走哪个仓、走哪条路。图里省略了鉴权和日志，因为它们不改变主业务判断。

不合格示例：
> 图中展示了从推单到出仓的完整流程，包括：推单接收、审单处理、库存匹配、出仓确认。

## 流程图精简规则

流程图和时序图中，以下步骤默认不画：
- 鉴权与权限校验
- 参数合法性校验
- 日志记录
- 序列化与反序列化
- 异常捕获与通用重试
- 基础数据库 CRUD 操作（除非这个操作本身有特殊业务含义）

每张流程图节点控制在 **5-8 个**。超过 8 个节点，通常说明图里混入了常识步骤，或者该拆成两张图。

## 生命周期写法规则

生命周期图或状态图节点控制在 **4-6 个**。

每个状态节点：
- 用业务术语命名，不用枚举值或类名
- 状态之间的迁移用业务动作触发，不用方法调用名

生命周期开头先说清三件事：
- 这个对象在什么情况下进入系统
- 它经历哪几个关键阶段
- 最后停在哪个稳定状态，或者为什么停不下来

## 模块对接写法规则

模块协作描述控制在 **4-6 个模块**。

每个模块：
- 用业务职责命名，不用技术包名或类名
- 只描述它负责什么、接谁的输入、把结果交给谁
- 不描述内部实现、子模块拆分、方法调用关系

## 判断句标准

每个大节第一句最好是判断句。合格的判断句通常至少包含以下之一：
- 这件事在业务上的核心特征是什么
- 数据或业务在这里发生了什么变化
- 为什么系统会这样组织
- 这个阶段对业务意味着什么

不合格示例：
- “这一节分析系统如何处理订单。”
- “下面从几个方面介绍相关逻辑。”
- “从代码实现上看，涉及多个模块。”

更好的写法：
- “出仓链的关键不在搬货，而在系统先决定这张单该走哪个仓。”
- “审单阶段是这条链的门槛：没过这一步，后面的匹配都不会发生。”
- “这三个模块不是三件独立的事，而是同一条业务链上的前后站。”

## 段落推进规则

一个好段落通常按这个顺序推进：
1. 先下判断
2. 再解释为什么
3. 再说明这意味着什么
4. 必要时补一句边界或例外

不是每段都必须写满四步，但至少要让读者感觉“这一段推进了理解”。

如果一个段落只是在：
- 搬运材料
- 罗列路径
- 解释你做了什么研究动作
- 重复上文已经说清的判断

那它应该被删除、压缩或后移。

## 节奏规则

- 开头句尽量短、硬、能立住
- 第一屏优先做成“可扫描”结构：一句主判断，三条短要点，一句读图提醒
- 正文每一节只回答一个核心问题，不要一节里塞多个平级主题
- 主线部分最厚，例外部分更短，证据部分最靠后
- 不要让每一节都一样长、一样重
- 当你发现自己在不断补免责声明，通常说明判断句还不够准

## 人性规则

读者在阅读深度研究时，心里通常会连续问这几件事：
1. 所以到底是什么
2. 为什么会这样
3. 什么时候不这样
4. 这对我有什么影响
5. 你凭什么这么说

好的报告要按这个顺序满足读者，而不是按作者的工作顺序写。

## 主线规则

- 正文必须有一条主线
- 所有小节都要服务这条主线
- 例外和补丁的任务是修正主线，不是抢走主线
- 如果一个信息点不改变主判断，就不要让它占据正文中心

## 列表和表格

只有在以下情况才优先使用表格：
- 明确做对比
- 明确做矩阵
- 明确做状态或阶段整理

只有在以下情况才优先使用列表：
- 需要压缩 3-5 个平级要点
- 需要让读者快速扫描关键项

其余情况优先段落。不要为了“看起来专业”而滥用表格和列表。

如果列表里每一条都很长，说明它其实在承担叙述功能，应当改回段落。

## 加粗与括号规则

加粗和括号都是视觉信号，滥用后信号会失效。

**加粗密度**：每个段落最多加粗 1-2 处。技术名词不计入配额，但本身也不必强制加粗。

**括号密度**：一句话里最多一处括号。括号能拆成短句的，优先拆开。

## 开头要求（意图自适应）

第一屏的任务是**立刻命中用户这次最想知道的答案**，具体写法随报告意图变：

- 事实确认型：先逐条给判定（是 / 不是 / 部分是 + 一句澄清）
- 决策支持型：先给可行性结论 + 最关键的前提或代价
- 机制解释型：先给“真正决定结果的是什么”
- 业务理解型：先给本质一句话 + 最值得先记住的几条
- 风险 / 对比 / 排查型：先给最该优先处理的风险 / 推荐选项 / 根因

通用要求：
- 不要把研究范围、证据来源、类型判定放在最前面
- 不要把第一屏写成均匀铺开的摘要段落
- 不要在第一屏堆太多风险、例外、免责声明
- **不要强行凑“三条判断”**——要几条写几条；有时第一屏只需要一个明确结论
- **只有业务理解型才适合“主结论 + 三条判断 + 读图提醒”那套**；其他意图不要套它

## 去 AI 腔规则（硬要求）

下列写法会让报告“一看就是 AI”，除非该意图真的需要，否则不要用：

- 标题不要导览腔：不要 `先给你…`、`会不会…`、`能做什么、不能指望什么`、`…各自是什么`、默认问句标题；标题默认用克制名词短语
- 不要把 `先看懂这件事`、`一张图先看整体` 当固定栏目名套到非业务理解报告上
- 不要每份报告都写“最值得先记住的三件事”再凑三条
- 不要每张图后都来一句“读图要点 / 图后说明 / 这张图帮你看清……”
- 不要每节开头写“本节分析…… / 下面从几个方面……”
- 不要满屏加粗、用 ✅❌ 符号堆表格当正文主力
- 不要把整篇写成"表格 + 列表 + 加粗"的拼盘，每节都要有散文骨架
- 不要每句话都挂“可能 / 据推断 / 理论上 / 暂时认为”

自检一：把正文具体内容抠掉，剩下的目录和框架如果放到任何项目都成立，就是套模板，必须重写。
自检二：把所有表格和列表删掉，剩下的散文如果读不成文，说明报告是拼盘不是文章，必须补散文骨架。

文风与审美的完整标准（标题、加粗、散文优先、循序渐进、节奏）见 `references/report-craft.md`，写最终报告前必须读。

## 技术注记规则

技术说明只允许以 `> 注：` 形式轻量穿插，规则如下：

- 格式固定：`> 注：[一到两行说明，不超过两行]`
- 内容用途：让有技术背景的读者快速回溯来源，不是解释实现细节
- 密度上限：正文每个大节最多 1-2 处技术注记
- 禁止发展成独立段落或独立技术分析节

合格示例：
```
> 注：该状态迁移由定时任务 SyncOrderJob 触发，每 5 分钟执行一次。
```

不合格示例：
```
> 注：SyncOrderJob 继承自 AbstractJob，通过 xxl-job 调度，配置在 xxl-job-admin 里，
> cron 表达式为 0 0/5 * * * ?，执行超时阈值为 120 秒，失败后会自动重试 3 次。
> 重试逻辑在 BaseRetryExecutor.executeWithRetry() 里实现。
```

## 视角要求

本技能面向技术读者，报告写作视角：
- 帮技术读者拿到他这次要的答案（一个判定、一个决策、一套机制的理解、或一张业务全景），而不是堆技术细节
- 代码、类、方法只在技术注记里出现，不进正文主叙事
- 报告不是代码说明书，也不是"换个项目也能用"的模板
- 报告该聚焦在哪（业务全景 / 事实判定 / 迁移可行性 / 风险排序…）由用户诉求决定，不固定为业务心智模型

## 完成标准

写完之后，至少检查以下八项：

1. **是否答到了**：对照蓝图 `Ask Map`，用户每个显式问题是否都被正面回答？
2. **第一屏是否命中**：第一屏是否直接给了用户最想要的答案，而不是先讲背景？
3. **起手是否够短**：每个大节开头是否已经立住判断，而不是先铺垫半天？
4. **标题是否安静**：节标题是否是克制的名词短语，而不是导览腔（"先给你…""会不会…""能做什么不能指望什么"）或套模板栏目名？详见 `references/report-craft.md`。
5. **是否套了模板**：抠掉具体内容后，框架是否还"放任何项目都成立"？如是，重写。
6. **图后说明是否从简**：有没有把图重说一遍？如有，删掉。
7. **技术注记是否失控**：是否有技术注记发展成独立段落？如有，压回两行以内。
8. **是否还在说黑话/套话**：是否仍有话术化表达或 AI 套话栏目？如有，换回平常说法。

## 删改规则

写完之后，至少做一次“删到更清楚”为目标的回看：
- 删掉不能回答“这意味着什么”的句子
- 删掉把正文写成研究日志的句子
- 删掉只是为了模板齐全而存在的小节
- 删掉那些既不改变主线、也不改变例外和风险判断的信息点
- 删掉对图、对上一段、对首屏判断的重复表达
- 把重复提醒、免责声明和证据口头禅压缩到一次

## 一句话原则

**好报告不是把材料摆得更全，而是贴着用户这次的问题把事情说清楚：诉求决定结构，答到问题先于模板完整，主线先于细节，短句先于包装词，平常话先于 AI 套话。**
