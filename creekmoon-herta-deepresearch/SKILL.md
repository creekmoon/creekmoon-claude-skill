# Herta 深度研究

> 这不是“更认真一点的普通分析”，而是一个强编排研究 skill。

这个 skill 的目标不是让一个代理查得更久，而是让主代理像研究负责人一样工作：
- 先把题目压成一个场景
- 先拆研究面，再拆子代理
- 先拿证据包，再拿结论包，再写报告
- 先处理例外和未确认，再给主结论

如果没有并行探索、没有证据分层、没有例外 / 风险 / 未确认，或者最终没有把这些压成读者更快看懂的结论，这次输出就还不算完成了深度研究。

进入本 skill 后，主代理只负责 6 件事：
- 判断这个任务是否值得进入深度研究
- 把题目收敛成一个可研究场景
- 拆研究面并派发首批子代理
- 基于探索结果追加定向研究和反向审查
- 合并冲突、降级结论、形成结论包
- 按读者模式产出最终报告

具体搜证规则、交付契约、写作模板和反模式，不写在本文件里，而是下沉到内部子路径。

## 触发策略

用户显式点名以下任一形式时，必须触发：
- `creekmoon-herta-deepresearch`
- “请用深度研究 skill”
- “请用这个 research skill”
- 通过 slash 命令、附加 skill、或其他明确方式指定本 skill

即使用户没有点名，只要同时满足“场景级问题”与“需要深挖”，也应主动触发。典型信号包括：
- 需要解释一个场景从输入到结果完整经历了什么
- 需要同时看代码、配置、文档、上下游关系中的两类或以上证据
- 需要分清主规则、例外规则、历史补丁、风险和未确认
- 需要产出研究报告，而不是一段快速回答
- 需要对业务口径与技术实现做交叉验证

以下情况不要触发：
- 单个类、方法、字段、表的局部解释
- 普通问答或快速结论
- 单文件调试、修 bug、查报错
- 不涉及场景闭环的零散问题

如果用户显式调用了本 skill，但题目太碎，先把题目收敛成一个场景问题，再启动正式研究。

## 默认研究姿态

一旦触发，就进入高强度深度研究：
- 优先场景闭环，不满足于局部解释
- 优先多面取证和交叉验证，不满足于单点证据
- 优先结论可靠性，不优先速度
- 研究层可以很重，但结论层必须更轻、更准、更容易被读者吸收
- 最终交付按“读者是否更快看懂关键判断”评分，而不是按“我做了多少工作”评分
- 只要环境支持且问题不是极小局部问题，默认必须先拆多个子代理并行探索
- 不要让一个子代理从头包办检索、研究、归纳和结论
- 优先“主代理统筹 + 探索子代理分流 + 研究子代理深挖 + 主代理统一收口”

如果环境支持 `Subagent`，主代理在正式派工前允许做的前置动作只有：
- 场景收敛
- 读取本 skill 的契约文件
- 最多一轮超轻量扫描，用来决定怎么拆研究面

如果主代理已经连续做了多轮搜索、读了很多文件、却还没有拆子代理，说明流程已经退化成普通分析，应立刻改回并行探索。

## 子代理硬约束

只要环境支持 `Subagent` 且问题不是极小场景级问题，就必须遵守：

1. 首批探索子代理应尽量在同一轮同时发出，而不是一个一个串行试探
2. 首批探索子代理至少 2 个，通常 3-4 个，高复杂度可到 4-6 个
3. 每个子代理只能负责一个明确切面，不能让两个子代理重复做同一件事
4. 探索子代理只交 `evidence-pack`，不交最终结论
5. 主代理不能把整个任务完整丢给某一个子代理
6. 如果因为环境限制不能起子代理，必须明确说明这次是降级执行，并按分面多轮法自己补齐

推荐首批切面：
- `logic-mainline`：主流程、阶段、关键判断、默认 / 回退
- `structure-config`：结构、配置、枚举、真正驱动行为的约束
- `relation-boundary`：参与方、职责边界、上下游依赖
- `failure-fallback`：失败路径、拒绝条件、补偿、兜底
- `docs-history`：文档、注释、配置来源、历史规则载体

## 临时工作区

当并行子代理过多、证据密度过高、或主代理判断“最后直接拼上下文会导致整合变差”时，允许并推荐使用临时工作区：
- 默认目录：`.herta-deepresearch-temp/`
- 可以按场景创建子目录，承载探索结果、证据包、合并笔记和结论底稿
- 子代理可以把中间产物写入该目录，主代理再基于这些文件做二次归并

这类临时文件不是正式交付物，而是为了降低最终收口时的信息爆炸。

## 主代理工作流

按下面顺序执行，不要跳阶段：

1. `Scene Framing`
   - 明确场景名称、主问题、起点、终点、主要参与方、输入输出、未覆盖范围
   - 判断主类型：`logic / structure / relation / mixed`
   - 判断读者模式：`business / technical / dual`
   - 如果预计会有高密度并行研究，先建立 `.herta-deepresearch-temp/{task}/00-scene-framing.md`

2. `Parallel Exploration`
   - 读取 `contracts/evidence-pack.md`
   - 优先调用 `agents/exploration-scout.md`
   - 按不同研究面拆出多个并行探索子代理，而不是先单代理长时间搜索
   - 探索阶段的目标是快速铺开证据地图、热点区域和后续深挖方向
   - 如果启用了临时工作区，探索结果应优先落到 `.herta-deepresearch-temp/{task}/exploration/`
   - 只有拿到 `Coverage`、`Hotspots` 和后续深挖建议后，才算探索阶段完成

3. `Focused Research`
   - 读取 `contracts/evidence-pack.md`
   - 基于探索阶段返回的热点区域，调用一个或多个研究子路径：
     - `agents/logic-tracer.md`
     - `agents/structure-mapper.md`
     - `agents/relation-analyst.md`
     - `agents/gap-auditor.md`
   - 研究子路径只允许返回证据包，不允许直接写最终报告
   - 至少有一轮研究要主动找反向证据、边界条件或失败路径来压测前面的判断
   - 如果启用了临时工作区，研究子路径应优先把 `evidence-pack` 写到 `.herta-deepresearch-temp/{task}/evidence/`

4. `Synthesis`
   - 读取 `contracts/conclusion-pack.md`
   - 调用 `agents/synthesizer.md`
   - 把证据包压缩成结论包：主线、关键规则、例外、风险、未确认项
   - 结论包不只是“筛过的材料”，还要给出第一屏骨架、正文展开顺序和证据后移建议
   - 当证据过多时，主代理应先在 `.herta-deepresearch-temp/{task}/merge/` 写去重和冲突矩阵，再进入归纳
   - 归纳阶段的任务是“选”和“压”，不是把所有发现原样搬过去

5. `Writing`
   - 读取 `contracts/report-contract.md`
   - 根据读者模式选择写作子路径：
     - `agents/writer-business.md`
     - `agents/writer-technical.md`
     - `agents/writer-dual.md`
   - 写作阶段只能使用已经稳定的结论包；除非发现关键冲突，不得重新发散搜证
   - 写作阶段不是把结论包排进模板，而是把主判断、机制、边界和证据尾部组织成更容易阅读的路径
   - 每个章节必须先写散文骨架（2-4 句判断 + 为什么 + 意味着什么），再决定是否用列表辅助；只有列表没有散文的节视为未完成
   - 第一屏优先用"1 段散文立住主判断 + 3 条关键要点 + 1 组例外 / 风险提示"的扫描结构
   - 最终报告必须先给判断，再给展开，再给证据尾部；如果第一屏还抓不住结论，说明写作尚未完成
   - 节标题必须是读者视角的导航判断句，不能是写作操作备注
   - 写完后必须按对应 writer-*.md 里的"写完后格式自检"做逐节检查
## 并行探索默认策略

“深度研究”在本 skill 里首先意味着：**默认不要单线程检索。**

默认拆分策略：

| 场景复杂度 | 默认并行子代理数 | 要求 |
|------|----------------|------|
| 极小但仍属场景级问题 | 2-3 个 | 至少拆开主流程面和结构 / 关系面 |
| 中等复杂度 | 3-4 个 | 默认拆开逻辑、结构、关系，并补一个失败 / 文档面 |
| 高复杂度或边界模糊 | 4-6 个 | 先广搜铺图，再按热点区域追加定向研究 |

优先拆分的研究面：
- 逻辑主线
- 结构 / 配置 / 枚举
- 关系 / 边界 / 上下游
- 失败路径 / 默认 / 回退
- 文档 / 配置载体 / 规则来源

只有在以下情况，才允许不做并行探索：
- 环境不支持子代理
- 问题极小，且确实不值得拆分

即使如此，也要明确说明为什么这次没有拆，以及你如何用分面轮次补偿这次降级。

## 深度研究完成标准

只有同时满足以下条件，才算真的做完了一次深度研究：
- 已说明这个场景的主线是什么，而不是只罗列搜到的点
- 已明确 3-5 条决定性规则，而不是只给“可能有关”的线索
- 已区分主规则、例外规则、历史补丁和风险点
- 已明确写出 `未确认`，而不是把缺证据的内容硬写成结论
- 核心判断至少经过两类证据面的交叉验证，或已解释为什么当前无法交叉验证
- 开头第一屏已讲清本质、主判断、关键例外 / 风险，以及这对当前读者意味着什么
- 正文按主线推进，而不是按研究过程、栏目清单或模板填空推进
- 最终交付不是研究台账，而是带判断、带节奏、能让人更快看懂的报告

如果输出里只有“文件路径 + 调用链 + 若干猜测”，这仍然不算深度研究。

## 主类型判断

主类型只决定“先看什么”，不决定最终必须写成什么样。

| 类型 | 核心问题 | 优先调用 |
|------|----------|----------|
| `logic` | 它什么时候触发、怎么判断、怎么流转、失败会怎样 | `logic-tracer` |
| `structure` | 它由什么组成、哪些字段 / 配置真正约束行为 | `structure-mapper` |
| `relation` | 谁参与、谁负责、上下游如何约束 | `relation-analyst` |
| `mixed` | 单一视角解释不完整 | 明确一个主类型，再补 1-2 个次类型 |

判断规则：
- 主要在问运行、判断、流转、回退，主类型偏 `logic`
- 主要在问字段、配置、枚举、结构约束，主类型偏 `structure`
- 主要在问参与方、职责边界、上下游依赖，主类型偏 `relation`
- 如果是 `mixed`，必须明确一个主类型，不能平均用力

## 读者模式判断

研究只做一套，表达按读者切换。

| 模式 | 适用对象 | 选择的写作子路径 |
|------|----------|------------------|
| `business` | 业务、产品、运营、非技术管理者 | `writer-business.md` |
| `technical` | 技术负责人、架构师、核心开发 | `writer-technical.md` |
| `dual` | 需要同时交付业务与技术两层视图 | `writer-dual.md` |

默认规则：
- 未指定读者时，先判断是否真的需要双视图
- 能用单视图说清的，优先单视图
- 只有在业务和技术两类读者都明确存在时，才使用 `dual`

## 强约束

- 结论强度不能超过证据强度
- 命名、注释、单个字段、局部片段只能算线索，不能单独当定论
- 高风险结论必须更保守，证据不足时宁可写 `未确认`
- 子代理只能交付自己阶段的产物，不能越界
- 主代理必须做统一收口，不能把整个任务完整丢给某一个子代理
- 主代理默认先拆检索面，再决定谁深挖；不要自己先埋头长搜
- 如果关键判断没有经过失败路径、反向证据或边界条件压测，默认不算稳定

## 阶段边界

为了避免注意力分散，必须遵守以下边界：
- 探索阶段先铺证据地图，不要过早进入结论写作
- 搜证阶段不要关心最终章节标题
- 归纳阶段不要为了凑格式制造结论
- 写作阶段不要发明新事实
- 主代理不要把研究过程暴露成读者的阅读顺序

- [ ] 已把题目收敛成一个场景问题
- [ ] 已明确主类型与读者模式
- [ ] 如果环境支持子代理且问题不是极小场景，已先拆出首批并行探索子代理
- [ ] 探索阶段已经产出热点区域和后续深挖方向
- [ ] 至少有一轮研究主动做了反向证据、失败路径或边界压测
- [ ] 当信息量过大时，已使用 `.herta-deepresearch-temp/` 落盘中间态后再合并
- [ ] 已先拿到证据包，再进入结论包
- [ ] 已先拿到结论包，再进入最终写作
- [ ] 写作阶段没有重新发散搜证
- [ ] 最终报告先给判断，再给展开，再给证据
- [ ] 所有高风险结论都经过降级审查
- [ ] 证据包里的每条候选规则都包含触发条件、对结果的影响和限制（不只有规则描述）
- [ ] 证据包里的每个热点都说明了"若不继续查会造成什么误判"
- [ ] 结论包里每条决定性规则都提供了机制草稿和后果草稿（不只有标签）
- [ ] 最终报告每个章节都有散文骨架，没有只有列表没有散文的节
- [ ] 最终报告节标题是读者视角的导航判断句，不是写作操作备注

## 一句话原则

**复杂场景要主动进深度研究；进入后先并行铺图，再定向深挖，最后由主代理统一收口。**
