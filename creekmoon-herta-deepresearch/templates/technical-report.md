# Technical Report Template

技术版报告的目标不是列出你看过哪些代码，而是让技术读者尽快知道：
- 系统真正靠什么规则决定结果，以及这几条规则背后的设计逻辑是什么
- 关键边界和例外在哪里
- 哪些默认值、回退和危险改动点最值得警惕

## 模板使用原则

- **标题全部是示例**，必须改写成对应场景的读者导航判断句。例如"三条过滤规则决定了哪些易仓订单能写入本地视图"，而不是"主线：系统通常如何决策"。
- **先翻译，再展开**：第一屏和各章开头先把复杂实现翻译成读者能直接复述的判断，再把类名、方法名、字段名挂回正文。
- **散文骨架优先**：每个章节先写 2-4 句散文（决策是什么 + 为什么这样设计 + 影响什么），再决定是否补列表或代码。
- **决策链而非执行链**：每节先说"真正决定结果的是哪条规则"，再说"这条规则是怎么实现的"，而不是先列调用链。
- **配置、锚点、研究动作默认后移**：`配置与运行环境锚点`、`证据锚点索引`、`研究方法说明` 这类内容只有在它们会改写结论时才值得进入正文，否则后移到证据尾部或删除。
- **如果某节没有真实内容，整节删除，不要保留空壳标题**。

```markdown
# [场景名称] 深度研究报告

## 1. [一句话主判断作为标题，例如："三条判断规则决定了库存匹配结果，现货优先于在途，同区优先于跨区"]

[不要把第一屏写成均匀铺开的摘要，优先使用下面这组固定槽位。首屏是给读者抓方向，不是给作者压缩正文。]

**[把主判断写完整，一句话，讲清系统在这个场景里的决策本质。先说真正决定结果的是什么，再说它牺牲了什么或带来了什么代价。]**

最值得先记住的三件事：

- [规则一的核心判断：读者能默念出来的短句，不超过 1.5 行，不嵌代码，不加括号，不把字段名/方法名直接塞进来。]
- [规则二的核心判断：同上。]
- [规则三的核心判断：同上。]

关键例外与危险改动点：

- [只保留 1-2 个最会改写主线或维护动作的点。]

[一句话：如果你现在要继续排查，先看哪条链路、哪类默认值或哪个边界。]

---

## 2. [研究边界本身会影响判断时再保留，例如："这份结论覆盖拉单到入镜像闭环，但不覆盖外部接口 SLA"]

[本章只在研究边界会影响主判断时保留。先用 2-4 句散文解释：这次结论覆盖到哪里、没覆盖哪里、这些边界会不会改变主判断。不要把本章写成研究日志。]

| 项目 | 内容 |
|------|------|
| 研究问题 | [这次研究要回答什么，用研究问题清单格式效果更好] |
| 起点 | [入口是什么，从哪里开始分析] |
| 终点 | [分析边界在哪里] |
| 主要参与方 | [服务/组件/表，说明各自职责] |
| 证据来源 | [代码、配置、文档等，只保留会影响可信度判断的来源] |
| 未覆盖范围 | [明确说出没查清楚的部分，以及这会不会影响主判断] |

---

## 3. [技术层面的主判断，例如："真正决定结果的不是调度入口，而是时间游标、单页上限和过滤规则"]

[先用 2-4 句散文建立这一章的主判断，再展开细节。读者看完这一段，应当已经知道本章最想讲的决策链是什么。]

### 3.1 [责任边界的判断句，例如："入口负责触发，但真正的决策集中在同步核心方法里"]

[先写散文骨架：谁是决策者，谁是执行者，谁承担后果，责任边界在哪里。再列参与方和链路。]

### 3.2 [真正决定结果的规则，标题改成规则内容，例如："时间游标、过滤规则和排序共同决定哪批订单会进入本地镜像"]

[先写散文骨架：最关键的那条规则是什么，为什么改变它结果就会不同。再展开决策链，不要用执行顺序流水账。]

### 3.3 [关键链路与状态流，标题改成链路的核心特征，例如："真正危险的不是拉取动作，而是写库前那段会改写更新时间的补丁逻辑"]

[先写散文骨架：链路里最关键的节点是哪个，为什么这个节点最容易引起问题。再展开链路图或状态流。]

### 3.4 [默认值、回退和危险改动点，标题改成具体风险，例如："单页上限和补拉分支是最容易被忽略的危险改动区"]

[先写散文骨架：最容易被忽视的隐性假设是什么，在什么场景下会失效。再列危险改动点。]

---

## 4. 风险与未确认（没有真实内容可整节删除）

### 4.1 [当前最值得提醒的风险，标题改成具体风险，例如："大批量窗口下的单页上限会制造漏单风险"]

[先写散文骨架：这个风险为什么存在（设计上的权衡），在什么条件下触发，触发后会影响什么。再说缓解方式。]

### 4.2 [仍会影响判断的未确认，标题改成具体疑点，例如："某条补拉分支是否在线上真实触发仍未证实"]

[只写那些如果判断错了，会改写主结论的缺口。说明缺口在哪里，以及判断错误会造成什么后果。]

---

## 5. 证据尾部（仅在需要增强可信度时保留）

### 5.1 [研究边界的短判断，例如："主结论不依赖外部平台文档，但部分接口行为仍待线上抓包确认"]

[简要说明没覆盖什么，以及这会不会影响主判断。]

### 5.2 [证据锚点的判断句，例如："主结论主要由同步核心方法、请求封装和下游消费 SQL 三处证据支撑"]

[只列核心结论需要的锚点，格式：结论 | 证据强度 | 锚点 | 仍缺什么。不要把正文重新抄一遍。]
```

## 散文骨架参考

合格的一节散文骨架示例（技术视角）：

> 真正决定最终发货仓库的不是入口服务，而是 `InventoryPoolSorter` 里的得分算法。每个库存池会被计算一个综合得分，得分最低的池子优先被消耗。改变任何一个加分系数，都会改写整批订单的发货结构，因此这个算法是危险改动区。

不合格示例（执行链流水账）：

> 1. 调用 `OfcOrderVerifyServiceImpl.processVerify()`
> 2. 调用 `InventoryPoolSearchServiceImpl.searchAvailableProductInventoryPool()`
> 3. 调用 `InventoryPoolSorter.sort()`
> 4. 返回排序后的库存池列表
> 5. 按顺序消耗直到满足订单需求

## 退稿自检

写完后，至少用下面四个问题卡一遍自己：

1. 第一屏三条 bullet 里是否出现了反引号、字段名、方法名、括号，或者任何一条已经长到像正文？只要有，就重写第一屏
2. 是否还保留了"配置与运行环境锚点""研究方法说明""关键锚点索引"这类泛标题，并把它们放在正文中心？只要有，就后移或删除
3. 是否存在某一节拿掉表格后就讲不清因果？如果是，说明这节还缺散文骨架
4. 读者看完前半部分后，能复述的是"哪几条规则决定结果"，还是只能复述"执行顺序"？如果是后者，说明全文仍然太像台账
