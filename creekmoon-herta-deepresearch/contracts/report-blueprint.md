# Report Blueprint Contract

`report-blueprint` 是报告层的第一份正式产物，也是 `final-report` 的唯一写作底稿。

由 `report-composer` 生成，写入 `.herta-deepresearch-temp/{task}/45-report-blueprint/report-blueprint.md`。

## 定位

蓝图不是第二份报告，也不是研究的延续。

它只做一件事：**先判定这次报告该是什么样，再把研究结论排成"贴合本次诉求、好直接成稿"的目录和底稿。**

和旧版"业务整形"最大的区别：蓝图不预设报告长成业务叙事的样子。报告长什么样，由用户诉求决定，蓝图负责把这个判断显式写下来。

## 强约束

- 只能由 `report-composer` 生成
- 只能基于已落盘的 `conclusion-pack`、`00-scene.md` 和 `evidence/` 重组，不引入新研究事实
- `Unconfirmed` 只能用于判断"什么不能进报告"和"研究边界怎么写"，不得改写成正文内容
- 它是底稿，不是成稿；保持短、可直接展开

## 输入

- `40-synthesis/conclusion-pack.md`
- `00-scene.md`（重点取里面的用户原始诉求和问题清单）
- 必要时回看 `20-evidence/`

## 固定输出结构

```markdown
## Intent Read（意图判定）
用户原始诉求一句话复述： [不加工，贴近用户原话的意思]
报告范式： [事实确认 / 决策支持 / 机制解释 / 业务理解 / 风险评估 / 对比选型 / 排查复盘 / 混合（说明主次）]
判定理由： [一句话：为什么是这个范式]
用户读完最想拿到的东西： [一个判定 / 一个可行性结论+改动清单 / 一张全景图 / 一份风险排序 / ...]

## Ask Map（问题映射）
把用户的每个显式问题列出来，并指明报告里哪一节正面回答它。
| 用户显式问题 | 直接答案（一句话先答） | 在报告哪一节展开 |
|--------------|------------------------|------------------|
| [问题一] | [是 / 不是 / 有条件，附一句关键澄清] | [章节名] |
| [问题二] | ... | ... |

## Derived Outline（推导目录）
本次报告的章节目录，按"用户最想先知道的"排序。每节写清它存在的理由。
标题用克制的名词短语（如"迁移可行性与改造范围"），不要导览腔（"先给你…""会不会…""能做什么不能指望什么"），详见 `references/report-craft.md`。
1. [章节标题（克制名词短语，准确命名这节讲什么）] — 这一节回答： [对应哪个问题/主线]
2. [章节标题] — 这一节回答： [...]
3. ...
（章节数量按需，事实确认型可能 3-4 节，业务理解型可能更多。没有对应问题或主线的章节不要出现。）

## First Screen（第一屏底稿）
第一屏要直接回答用户最核心的问题，不写"这件事是什么"的背景铺垫。
- 最核心结论： [一句话，直接命中用户最想知道的那个判断]
- 其余要点： [几条就写几条，不强行凑三条；没有就只写核心结论]
- 是否需要读图提醒： [需要才写；纯文字/表格型报告通常不需要]

## Section Briefs（各节底稿）
对 Derived Outline 里每一节，给一句核心判断 + 要点，供主体直接展开。
### [章节一标题]
核心判断： [这一节要立住的那句话]
要点： [2-4 条，或一段话]
### [章节二标题]
...

## Visual Plan（图示计划，按需）
这次报告值得画几张图就写几张；一张都不需要时写"本次不需要图示"，并说明为什么文字/表格已足够。
| 图名 | 这张图帮读者先看懂什么 | 图型 | 关键元素 | 证据锚点 |
|------|------------------------|------|----------|----------|

## Report Boundary（研究边界）
未覆盖范围： [这次没展开什么]
是否影响主判断： [不影响 / 有条件影响，说明原因]
（只写覆盖范围与影响，不转抄待验证问题。）

## Anti-AI Notes（去 AI 腔提醒）
本次成稿时特别要避免的套话/模板腔： [针对本次报告列 1-3 条，例如"不要套用全景图栏目""不要凑三件事"]

## Trace Map（回溯）
| 正文主判断 | conclusion-pack 来源 | 证据锚点 |
|------------|----------------------|----------|
| [判断一] | [规则 N / Decisive Rules / ...] | [文件+行] |
```

## 压缩与质量标准

- `Intent Read` 必须明确写出范式，不能含糊（含糊 = 没判意图）
- `Ask Map` 必须覆盖用户每个显式问题，且每个问题都有"在哪节回答"
- `Derived Outline` 的每一节都要能说出存在理由；说不出就删
- `First Screen` 的最核心结论必须直接命中用户诉求，不是泛泛的"这件事是什么"
- `Visual Plan` 允许为空（写明本次不需要图），不强制凑图
- 整份蓝图更像提纲短底稿，不是可直接发的成稿

## 禁止项

- 不要不判意图就默认套业务理解骨架
- 不要让 `Derived Outline` 变成"换个项目也能用"的固定模板目录
- 不要漏掉用户的显式问题
- 不要为了"显得完整"强行加生命周期、模块对接、为什么这样组织等用户没问的节
- 不要强行凑"最值得先记住的三件事"
- 不要把 `Unconfirmed` / `待验证` 改名后塞进任何一节
- 不要引入未经研究证实的新事实

## 一句话原则

**蓝图先把"这次报告该长什么样"判断清楚并写下来，再把研究结论排成贴合本次诉求的目录和短底稿，让最终报告能直接写成"专门为这次问题写"的样子。**
