# Report Shaping Pack Contract

`shaping-pack` 是报告整形层的唯一正式产物，也是 `final-report` 的唯一写作底稿。

## 定位

报告整形层不是继续研究，也不是开始成文。

它做的是：把已经完成的研究结论，重组成“主 agent 很容易写、读者很容易看懂”的短底稿。

**整形层的唯一使命：把研究结果整理成好写、好读的业务叙事顺序，而不是再写一份准报告。**

## 强约束

- `shaping-pack` 只能由 `business-architecture-shaper` 或主 agent 生成
- 必须写入 `.herta-deepresearch-temp/{task}/45-report-shaping/shaping-pack.md`
- 只能基于已落盘的 `conclusion-pack` 和 `evidence/` 文件进行重组，不引入新的研究事实
- 它不是最终报告；它只是最终报告的短底稿
- 若某个维度在研究阶段未能证实，此处同样不能凭空填入

## 输入

整形层必须读取：
- `40-synthesis/conclusion-pack.md`
- `00-scene.md`
- 必要时回看 `20-evidence/` 各文件，补充流转、生命周期或模块协作细节

## 压缩标准

- `Opening Frame` 只保留真正会改变读者理解的三条判断，不写功能介绍
- `Business Panorama` 只交代“这件事整体长什么样”，不要展开成正文段落
- `Flow Spine` 只稳定主线：起点、转折、终点、关键节点
- `Lifecycle Spine` 只有在它确实改变理解时才单独保留
- `Module Handoff` 只有在模块分工是读者理解主线所必需时才单独保留
- `Why This Shape` 只保留真正解释组织形态的原因，不做全量总结
- `Light Anchor Notes` 每处注记不超过两行，不能展开成段
- 整份 `shaping-pack` 应更像提纲和短底稿，而不是可以直接发给读者的成稿

## 图示规划原则

每张图都要回答“这张图要帮读者先看懂什么”，但答案尽量短，最好一句话说完。

图型选择指引：
- 关键对象与边界全景 → 关系图或分层图
- 数据/业务流转路径 → 精简流程图或时序图
- 业务对象状态变化 → 状态图
- 模块职责与协作 → 协作图或责任边界图

约束：
- 每张图节点数量有上限：关系图 ≤ 8，流程图节点 ≤ 8，状态图 ≤ 6，模块图 ≤ 6
- 图不应比正文更复杂；如果需要很多话才能解释图，说明图画得太细
- 图省略的内容只在必要时说明，避免为了“交代完整”而把说明写长

## 禁止项

- 不要引入未经研究阶段证实的新结论或新事实
- 不要把 `shaping-pack` 写成最终报告
- 不要把规则清单原样搬进 `Opening Frame`
- 不要把 `Module Handoff` 写成类/方法调用链
- 不要为了让图“看起来完整”而把众所周知步骤画进去
- 不要在 `Light Anchor Notes` 里发展出技术分析段落
- 不要使用黑话、包装词或互联网公司话术把短底稿写重

## 一句话原则

**`shaping-pack` 不是第二份报告，而是给主 agent 的短底稿：把研究结果按最容易讲清楚的顺序排好，让最终报告能直接写成平常话。**
