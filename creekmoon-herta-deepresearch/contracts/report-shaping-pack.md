# Report Shaping Pack Contract

`shaping-pack` 是报告整形层的唯一正式产物，也是 `final-report` 的唯一写作底稿。

## 定位

报告整形层不是继续研究，也不是开始成文。

它做的是：把已经完成的研究结论，重新组织成"能帮助技术人员理解业务"的叙事顺序和图示规划，让 writer 几乎可以直接按这份底稿展开报告。

**整形层的唯一使命：让读者读懂业务，而不是让读者看清代码。**

## 强约束

- `shaping-pack` 只能由 `business-architecture-shaper` 或主 agent 生成
- 必须写入 `.herta-deepresearch-temp/{task}/45-report-shaping/shaping-pack.md`
- 只能基于已落盘的 `conclusion-pack` 和 `evidence/` 文件进行重组，不引入新的研究事实
- 不是最终报告；它是最终报告的业务化叙事底稿
- 若某个维度在研究阶段未能证实，此处同样不能凭空填入

## 输入

整形层必须读取：
- `40-synthesis/conclusion-pack.md`
- `00-scene.md`
- 必要时回看 `20-evidence/` 各文件，补充流转、生命周期或模块协作细节

## 固定输出结构

```markdown
## Opening Frame
一句话讲清这件事是什么（业务语言，不出现类名/方法名）。
技术人员为什么需要先理解它。
第一屏最该记住的三件事（只保留改变理解的判断，不列规则清单）：
- [判断一]
- [判断二]
- [判断三]

## Business Panorama
这件事整体长什么样。
关键对象：[谁参与其中]
上游：[谁提供输入]
下游：[谁消费结果]
关键边界：[哪里是系统边界或业务责任边界]
建议图型：[关系图 / 分层图 / 总览流程图]
图示规划：[这张图要帮读者先看懂什么，哪些元素必须出现，哪些省略]

## Flow Spine
数据和业务从哪里开始流动，经过哪些关键节点，最终落到哪里。
省略说明：以下众所周知步骤不进入图示——[例：鉴权、参数校验、日志、序列化]
建议图型：[精简流程图 / 时序图]
关键节点（控制在 5-8 个）：
- [节点一：说明这个节点的业务意义，不是技术动作]
- [节点二：同上]
图示规划：[这张图要帮读者先看懂什么]

## Lifecycle Spine
核心业务对象经历哪些关键状态或生命周期阶段。
每个阶段代表什么业务变化（不是实现动作）：
- [阶段一 → 阶段二]：[发生了什么业务变化]
- [阶段二 → 阶段三]：[同上]
建议图型：[状态图 / 生命周期时序图]
图示规划：[这张图要帮读者先看懂什么]
是否需要单独成图：[是 / 否，理由]

## Module Handoff
哪些模块在承接、加工、输出这件事。
每个模块只写职责与对接，不写类名、方法名、字段名：
- [模块 A]：负责 [业务职责]，接收来自 [谁]，输出给 [谁]
- [模块 B]：同上
建议图型：[协作图 / 责任边界图]
图示规划：[这张图要帮读者先看懂什么]
是否需要单独成图：[是 / 否，理由]

## Why This Shape
为什么系统被组织成今天这样。
背后的业务诉求是什么。
有哪些架构约束或历史因素。
如果用另一种方式组织，会出现什么问题。
（这部分用来撑起报告的"为什么"一节，不是研究结论本身）

## Light Anchor Notes
正文哪些位置可以出现轻量技术注记（用 `> 注：` 形式），以及注记的粒度上限。
每条注记的目标是"让有背景的读者快速回溯来源"，而不是"解释实现细节"。
示例：
- 在"数据流第三节点"处：`> 注：该同步由定时任务 SyncOrderJob 承接，每 5 分钟执行一次。`
- 在"生命周期第二阶段"处：`> 注：状态由 OrderStatusEnum.AUDITING 变更为 OrderStatusEnum.DISPATCHING。`

## Trace Map
| 正文判断 | shaping-pack 来源 | conclusion-pack 条目 | 证据锚点 |
|----------|-------------------|----------------------|----------|
| [判断一] | [Flow Spine / Lifecycle / ...] | [规则 N] | [文件+行] |
| [判断二] | 同上格式 | | |
```

## 压缩标准

- `Opening Frame` 只保留真正会改变读者理解的三条判断，不写功能介绍
- `Business Panorama` 的关键对象控制在 4-7 个，不要把所有参与方都列进来
- `Flow Spine` 的关键节点控制在 5-8 个，常识步骤必须省略
- `Lifecycle Spine` 的关键阶段控制在 4-6 个，中间过渡阶段合并
- `Module Handoff` 的模块控制在 4-6 个，内部子模块不展开
- `Why This Shape` 只保留真正解释组织形态的原因，不做功能全貌描述
- `Light Anchor Notes` 每处注记不超过两行，不能展开成段

## 图示规划原则

每张图必须回答"这张图要帮读者先看懂什么"，不能只指定图型名称。

图型选择指引：
- 关键对象与边界全景 → 关系图或分层图
- 数据/业务流转路径 → 精简流程图或时序图（跳过常识步骤）
- 业务对象状态变化 → 状态图（只保留关键状态迁移）
- 模块职责与协作 → 协作图或责任边界图（不画调用细节）

约束：
- 每张图节点数量有上限：关系图 ≤ 8，流程图节点 ≤ 8，状态图 ≤ 6，模块图 ≤ 6
- 图不应比正文文字更复杂，如果需要大量文字才能解释图，说明图画得太细
- 图省略的内容必须在"图示规划"栏说明，避免读者误以为系统只有图里这些东西

## 禁止项

- 不要引入未经研究阶段证实的新结论或新事实
- 不要把 `shaping-pack` 写成最终报告
- 不要把规则清单原样搬进 `Opening Frame`
- 不要把 `Module Handoff` 写成类/方法调用链
- 不要为了让图"看起来完整"而把众所周知步骤画进去
- 不要在 `Light Anchor Notes` 里发展出技术分析段落

## 一句话原则

**`shaping-pack` 的任务不是整理研究结果，而是把研究结果翻译成业务理解顺序——让 writer 拿到这份底稿后，能直接写出帮助技术人员理解业务的报告，而不需要再去猜"先讲什么"。**
