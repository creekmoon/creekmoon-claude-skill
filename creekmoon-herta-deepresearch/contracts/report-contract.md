# Report Contract

`final-report` 是对外交付的唯一正式研究报告，由 `report-composer` 撰稿官产出，主 agent 负责把关与量化校验。

它的结构不是固定的——结构由这次报告的意图决定（见 `references/report-archetypes.md`）。本契约约束的是「不管哪种意图都要守的写作标准」；报告的文风与审美（标题腔、加粗、散文 vs 表格、循序渐进）见 `references/report-craft.md`，写作前必须一并遵守。

## 强约束

- 由 `report-composer` 产出，写入 `.herta-deepresearch-temp/{task}/50-report/final-report.md`
- 只能基于 `45-report-blueprint/report-blueprint.md` 写作，不直接展开 `conclusion-pack`，也不再发散搜证
- 报告阶段不引入蓝图里没有的新事实
- 不把研究过程、搜索过程、文件导览写成正文主体
- 不设 `To Verify` / `未确认` / `待验证` 栏目
- 主 agent 不接受"换个项目也能套用"的模板报告，发现就退回重写

## 结构来自意图，不来自模板

写报告前，先确认蓝图里的 `Intent Read` 判定了哪种范式，然后：

- 目录直接用蓝图的 `Derived Outline`，它已经按本次问题推导好了
- 不要在 `Derived Outline` 之外，因为"模板惯例"再补全景图节、生命周期节、模块对接节、为什么这样组织节
- 各范式的建议骨架见 `references/report-archetypes.md`，那是参考不是必填

只有当 `Intent Read` 判定为「业务理解型」时，才适用"先看懂这件事 → 全景图 → 流转 → 生命周期 → 模块对接 → 为什么这样组织"这套骨架。其他范式不要套它。

## 所有范式共用的写作底线

- **第一屏直接回答用户最核心的问题**，不先讲"这件事是什么"的背景
- **用户每个显式问题，正文都能找到正面、明确的回答**（对照蓝图 `Ask Map` 逐条核）
- 全文有一条主线，所有小节服务这条主线
- 围绕"用户要的判断/结论"写，不围绕代码、类、方法写
- 图按需出现：能帮理解就画，文字/表格说得清就别硬画
- 技术说明只做轻量注记，不发展成独立技术段落
- 最后给证据回溯，保证可复核

## 第一屏要求（意图自适应）

第一屏的任务是**立刻命中用户最想知道的那个答案**，写法随意图变：

- 事实确认型：先逐条给判定（是 / 不是 / 部分是 + 一句澄清）
- 决策支持型：先给可行性结论 + 最关键的前提或代价
- 机制解释型：先给"真正决定结果的是什么"
- 业务理解型：先给本质一句话 + 最值得先记住的几条
- 风险/对比/排查型：先给最该优先处理的风险 / 推荐选项 / 根因

通用要求：
- 第一屏不堆研究范围、证据来源、类型判定
- 不强行凑"三件事"；要几条写几条
- 第一屏不出现类名、方法名、枚举值、接口路径（Kafka、MongoDB 这类用户本来就在问的产品名可以出现）

## 直白表达要求

- 正文优先用平常说法，先把意思说清楚，再考虑文采
- 可以用必要技术词
- 不要把互联网公司话术当主叙事（扇入、扇出、闭环、抓手、钩子、占位、赋能、收口）
- 不靠比喻、黑话、包装词撑句子；一句话离开这些词就说不清，说明该重写
- 首选短句

## 去 AI 腔（硬要求）

这是这份契约最重要的部分。下列写法会让报告"一看就是 AI 套模板"，除非该范式真的需要，否则禁止：

- 不要无脑用 `## 先看懂这件事`、`## 一张图先看整体` 当固定栏目名套到非业务理解报告上
- 不要每份报告都写"最值得先记住的三件事"再凑三条
- 不要每张图后都来一句"读图要点 / 图后说明 / 这张图帮你看清……"
- 不要满屏加粗（每段最多 1-2 处加粗）
- 不要用 ✅❌ 等符号堆成正文主力，表格里也克制使用
- 不要每节开头写"本节分析…… / 下面从几个方面……"
- 不要每句话都挂"可能 / 据推断 / 理论上 / 暂时认为"

**自检：把正文具体内容抠掉，剩下的目录和框架如果放到任何项目都成立，就是套模板，必须重写。**

## 图示要求（按需，不强制）

- 图的数量由这次研究本身决定，不为凑数而画；一张都不需要时就不画
- 要画时，图型按意图选：
  - 对象关系与边界 → 关系图或分层图
  - 数据/业务流转 → 精简流程图或时序图
  - 业务对象状态变化 → 状态图
  - 模块职责与协作 → 协作图或责任边界图
  - 多事实点的数据流向串联 → 精简流向图
- 节点上限：关系图 ≤ 8，流程图 ≤ 8，状态图 ≤ 6，模块图 ≤ 6
- 图中不出现类名、方法名、SQL、接口路径
- 流程图省略鉴权、日志、参数校验、序列化、通用重试、基础 CRUD 等常识步骤
- 图必须服务理解，不做装饰；图后说明默认从简，只在省略内容可能误导时补一句

## 技术说明格式

技术说明只允许以轻量注记穿插，格式固定：

```
> 注：[一到两行技术背景，让有背景的读者快速回溯来源]
```

禁止把技术注记展开成独立段落或独立技术分析节。

## 证据回溯

报告结尾固定给出：

- **研究边界**：没覆盖什么、影不影响主判断（只写覆盖范围与影响，不写待验证清单）
- **证据回溯表**：正文主判断 → 证据锚点，让读者能复核

```markdown
## 证据回溯与研究边界

### 研究边界
[没覆盖什么，影不影响主判断]

### 证据回溯表
| 正文主判断 | 证据锚点 |
|------------|----------|
| ... | ... |
```

## 写作要求

- 每节先用一段散文把核心讲清楚，再决定要不要上表格、列表、图（散文骨架优先）
- 同一意思只说一次：首屏说过的，后文不要换个说法再说
- 主线部分写厚，例外部分写短，证据部分最靠后
- 段落之间要有承接，让读者顺着一条线读下去，而不是面对一堆独立词条
- 写完对照蓝图 `Ask Map`，确认每个用户问题都被正面回答

## 标题与审美（硬要求，详见 `references/report-craft.md`）

报告的 AI 味最大来源是标题腔和拼盘式排版，本节是底线：

- **标题默认用克制的名词短语**，准确命名这节讲什么（如"迁移可行性与改造范围"）
- 标题禁止导览腔：不要第二人称（"先给你…"）、不要默认问句（"会不会…"）、不要卖关子并列（"能做什么、不能指望什么"）、不要口语动词（"怎么流""卡不卡"）
- 判断句标题是少用工具：全篇最多一两个，且不用问句、不用第二人称
- **加粗克制**：每段最多 1-2 处，只给真正该停下来记住的那一处
- **散文骨架优先**：每个主要章节都要有真正的散文，不能只有表格 / 列表 / 代码块
- **表格只在真对比、真清单、真矩阵时用**；能用一段话说清的，不要做成表
- 少用 ASCII 箭头图（`A → B → C`）；要么写成一句因果叙述，要么用规范 mermaid
- 不用助手腔（"我们可以看到""值得注意的是""综上所述"）、不用 ✅❌ 当正文主力

## 禁止项

- 不要不判意图就套用业务理解固定目录
- 不要写出"换个项目也成立"的通用模板报告
- 不要漏答用户的显式问题
- 不要直接基于 `conclusion-pack` 写报告，必须经过 `report-blueprint`
- 不要把类名、方法名、SQL 写成正文主叙事
- 不要把代码调用链写成正文段落
- 不要为模板完整保留空壳章节
- 不要为凑数而画图
- 不要把未确认内容写成确定结论，或改名后塞进研究边界

## 一句话原则

**最终报告要先认准用户这次到底要什么，让目录、第一屏、表达结构都为这个诉求服务；把事情用平常话说清楚、把每个问题正面答到、把证据补上，而不是套一份"放哪都行"的模板。**
