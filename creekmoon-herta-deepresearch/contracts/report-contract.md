# Report Contract

`final-report` 是主 agent 对外交付的唯一正式研究报告，也是读者真正会阅读的成稿。

## 强约束

- 最终报告只能由主 agent 生成
- 必须写入 `.herta-deepresearch-temp/{task}/50-report/final-report.md`
- 只能基于 `conclusion-pack` 和对应证据锚点写作
- 报告阶段不再发散搜证
- 不要把研究过程、搜索过程、文件导览写成正文主体
- 最终报告必须是对象中心、渐进展开的成稿，不是研究台账
- 最终报告必须包含图示；默认使用 Markdown 可落地的文本图，优先 `Mermaid`

## 固定目标

最终报告必须做到：
- 第一屏先让读者知道：这件事是什么、为什么值得看、最关键的判断是什么
- 让读者先建立全景，再进入链路、规则、例外和风险
- 围绕研究对象写作，而不是围绕作者看过哪些代码写作
- 用至少一张图帮助读者获得空间感、关系感或流转感
- 最后提供证据回溯，保证可复核

## 固定结构

```markdown
# 标题

## 先看懂这件事
**[一句话主判断。先讲真正决定结果的东西，再讲代价或风险。]**

最值得先记住的三件事：
- ...
- ...
- ...

如果你只准备继续排查或改动，先看：
- ...

## 一张图先建立全景
[必须放 1 张总览图。可选：关系图 / 分层图 / 流程总览图。]

[2-4 句散文：这张图想帮读者先建立什么感觉，哪些对象或边界最重要。]

## 对象入场：它是什么，为什么它是这条链的主角
[先用 2-4 句散文，把研究对象放回宏观场景：它扮演什么角色，为什么读者应先理解它。]

## 这条链平时是怎么运转的
[先用 2-4 句散文讲主线，再按场景放流程图 / 时序图 / 状态图。]

## 真正决定结果的规则
### [规则 1 的判断句]
[先写为什么这条规则存在，再写它怎样改写结果。]

### [规则 2 的判断句]
[同上]

### [规则 3 的判断句]
[同上；如确有必要，可扩到 4-5 条]

## 关键例外与危险断点
[先讲哪一两个地方最容易让读者误判，再讲这些例外如何打断主规则。]

## 风险与未确认
[只保留真正会改写当前判断的风险和未确认。若没有，也要明确写无。]

## 证据回溯表
| 主判断 | 对应结论项 | 证据锚点 |
|--------|------------|----------|
```

## 图示要求

- `一张图先建立全景` 是必填节，必须至少出现 `1` 张图
- 图型按场景选，不强行统一：
  - 关系复杂时优先关系图
  - 链路复杂时优先流程图或时序图
  - 状态变化关键时优先状态图
  - 分层与边界关键时优先分层图
- 若一个场景只需要 `1` 张图就能建立全景，不要为了凑数硬塞第二张
- 若主线必须依赖第二张图才能讲清，可以在 `这条链平时是怎么运转的` 中追加
- 图必须服务判断，不得只是装饰

## 写作要求

- `先看懂这件事` 必须短、准、直接，不要先铺背景，不要塞代码名
- `最值得先记住的三件事` 必须正好 `3` 条，每条都应是能被读者默念出来的短判断
- `对象入场` 必须先讲对象在宏观场景中的角色，再挂回类、表、接口、任务名
- `这条链平时是怎么运转的` 要按对象经历和结果发生顺序讲，不要按研究顺序讲
- `真正决定结果的规则` 要先讲规则为什么存在，再讲代码怎样实现
- `关键例外与危险断点` 要优先讲会打断主线、改变维护动作的点
- `风险与未确认` 要明确哪些内容还不能下强结论
- `证据回溯表` 必须覆盖报告中的主判断，不能留空锚点

## 禁止项

- 不要把 `主类型`、`读者模式`、`研究阶段划分` 写进报告
- 不要把路径列表、搜索关键词、工作量说明放在报告开头
- 不要只给结论不给依据
- 不要把未确认内容写成确定结论
- 不要让子代理代写最终报告
- 不要把图画成默认真实世界全貌；图只是当前证据支撑下的阅读辅助

## 一句话原则

**最终报告是主 agent 用证据写成的阅读作品：先让读者想看，再让读者看懂，最后让读者能够回溯。**
