# Report Contract

`final-report` 是主 agent 对外交付的唯一正式研究报告，也是读者真正会阅读的成稿。

## 强约束

- 最终报告只能由主 agent 生成
- 必须写入 `.herta-deepresearch-temp/{task}/50-report/final-report.md`
- 只能基于 `45-report-shaping/shaping-pack.md` 写作，不直接展开 `conclusion-pack`
- 报告阶段不再发散搜证，不再引入新的技术判断
- 不要把研究过程、搜索过程、文件导览写成正文主体
- 最终报告的目标是帮助技术人员看懂业务，不是展示技术细节
- 最终报告不设 `To Verify` / `未确认` / `待验证` 栏目

## 固定目标

最终报告必须做到：
- 第一屏先让读者知道：这件事到底是什么，为什么值得看，最该先记住哪三点
- 先建立整体业务形态，再进入数据和业务流转，再按需要讲生命周期或模块协作，最后解释为什么
- 围绕业务对象和业务流转写作，而不是围绕代码、类、方法写作
- 用图帮助读者建立整体印象，但图和图后说明都要短
- 技术说明只做轻量回溯，不发展成独立技术段落
- 最后提供证据回溯，保证可复核

## 直白表达要求

- 正文优先用平常说法，先把意思说清楚，再考虑文采
- 可以使用必要技术词，例如 Kafka、Topic、消费者、配置项、状态机
- 不要把常见互联网公司话术当正文主叙事，例如：`扇入`、`扇出`、`闭环`、`抓手`、`钩子`、`占位`、`赋能`、`收口`
- 不要靠比喻、黑话或包装词撑句子；如果一句话离开这些词就说不清，说明这句该重写
- 首选短句。能一句说完，就不要拆成三句铺垫

## 固定结构

```markdown
# [场景名称] 业务深度研究报告

## 先看懂这件事

**[一句话讲清这件事是什么。先说本质，不绕。]**

最值得先记住的三件事：
- [改变理解的判断一，不是功能介绍]
- [改变理解的判断二]
- [改变理解的判断三]

[一句话告诉读者先看哪张图，或先抓哪条主线。]

---

## 一张图先看整体

[先用 1-2 句交代：这张图帮读者先建立什么整体印象。]

```mermaid
[业务总览图：关系图 / 分层图 / 总览流程图，节点数 ≤ 8]
```

[图后只补 1-2 句：先看哪里；如果有必要，再补一句省略了什么。不要重复图内容。]

[先用 1-3 句说清：起点是什么，关键转折在哪里，终点是什么。]

[简要说明没覆盖什么，以及这会不会影响主判断。这里只写覆盖范围与影响，不写待验证问题、猜测性判断或补证清单。]

### 证据回溯表
| 正文主判断 | shaping-pack 来源 | 证据锚点 |

| ... | ... | ... |
```

## 图示要求

- `一张图先看整体` 是必填节，必须出现 1 张总览图
- 图型按业务理解需要选：
  - 对象关系与边界 → 关系图或分层图
  - 数据/业务流转 → 精简流程图或时序图
  - 业务对象状态变化 → 状态图
  - 模块职责与协作 → 协作图或责任边界图
- 每张图节点数量有上限：关系图 ≤ 8，流程图 ≤ 8，状态图 ≤ 6，模块图 ≤ 6
- 图中不出现类名、方法名、SQL 语句、接口路径
- 图必须服务理解，不得只是装饰
- 图后说明默认从简；只有省略内容可能让读者误解时，再补一句省略说明

## 技术说明格式

技术说明只允许以轻量注记形式穿插，格式固定为：

```
> 注：[一到两行的技术背景说明，让有背景的读者快速回溯来源]
```

禁止把技术注记展开成独立段落或独立技术分析节。

## 写作要求

- `先看懂这件事` 必须短、准、直接，三条判断都应是读者能直接复述的短句
- 流转节先说“从哪里开始，在哪一步转弯，最后落到哪里”
- 生命周期节先说“对象经历了哪几个关键阶段，每个阶段意味着什么”
- 模块节先说“谁承接、谁加工、谁输出”
- 每个大节先立住一个核心判断，不要求为了模板完整硬写 2-4 句铺垫
- 图前文字负责给读者定方向；图后文字负责补充关键理解，不负责复述图
- 节标题必须是读者视角的导航句，不是操作备注
- 同一意思只说一次：已经在首屏说过的，不要在后文换个说法再说一遍

## 禁止项

- 不要直接基于 `conclusion-pack` 写报告，必须经过 `shaping-pack`
- 不要把规则清单作为正文核心
- 不要出现类名、方法名、SQL 语句作为正文主叙事
- 不要把代码调用链写成正文段落
- 不要把模块协作写成类/方法调用关系
- 不要把众所周知的步骤画进图里
- 不要让技术注记发展成独立的技术分析段落
- 不要为了模板完整保留空壳章节
- 不要把未确认内容写成确定结论
- 不要把 `Open Questions` / `Unconfirmed` / `待验证` 改名后塞进最终报告
- 不要让子代理代写最终报告
- 不要把比喻、黑话、互联网公司话术写成主要表达方式

## 一句话原则

**最终报告要先把事情说清楚，再把证据补上：让技术人员能快速看懂业务是什么、怎么流、为什么这样组织，而不是让他们先去猜作者到底想说什么。**
