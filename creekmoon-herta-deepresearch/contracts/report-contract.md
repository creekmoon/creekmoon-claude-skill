# Report Contract

`final-report` 是主 agent 对外交付的唯一正式研究报告，也是读者真正会阅读的成稿。

## 强约束

- 最终报告只能由主 agent 生成
- 必须写入 `.herta-deepresearch-temp/{task}/50-report/final-report.md`
- 只能基于 `45-report-shaping/shaping-pack.md` 写作，不直接展开 `conclusion-pack`
- 报告阶段不再发散搜证，不再引入新的技术判断
- 不要把研究过程、搜索过程、文件导览写成正文主体
- 最终报告的目标是帮助技术人员理解业务，不是展示技术细节

## 固定目标

最终报告必须做到：
- 第一屏先让读者知道：这件事是什么、为什么值得看、三个最值得先记住的判断
- 先建立整体业务形态，再进入数据流转，再进入生命周期，再进入模块协作，最后解释为什么
- 围绕业务对象和业务流转写作，而不是围绕代码、类、方法写作
- 用图帮助读者建立业务空间感，图必须精简，跳过常识步骤
- 技术说明以轻量注记形式穿插，不发展成独立技术段落
- 最后提供证据回溯，保证可复核

## 固定结构

```markdown

## 先看懂这件事

**[一句话讲清这件事是什么，以及它在业务上的核心特征。]**

最值得先记住的三件事：
- [改变理解的判断一，不是功能介绍]
- [改变理解的判断二]
- [改变理解的判断三]

[一句话告诉读者接下来先从哪张图进入。]

[这一章必须有图。优先放总览图，帮读者先建立业务对象、参与方、上下游、结果和关键边界的空间感。]

```mermaid
[业务总览图：关系图 / 分层图 / 总览流程图，节点数 ≤ 8]
```

```mermaid
[精简流程图或时序图，只保留改变理解的关键节点，≤ 8 个节点]

[状态图或生命周期图，关键状态 ≤ 6 个]

## 图示要求

- `一张图先看整体` 是必填节，必须出现 1 张总览图
- 图型按业务理解需要选：
  - 对象关系与边界 → 关系图或分层图
  - 数据/业务流转 → 精简流程图或时序图
  - 业务对象状态变化 → 状态图
  - 模块职责与协作 → 协作图或责任边界图
- 每张图节点数量有上限：关系图 ≤ 8，流程图 ≤ 8，状态图 ≤ 6，模块图 ≤ 6
- 图中不出现类名、方法名、SQL 语句、接口路径
- 省略的常识步骤必须在图后说明中提及
- 图必须服务理解，不得只是装饰

## 技术说明格式

技术说明只允许以轻量注记形式穿插，格式固定为：

```
> 注：[一到两行的技术背景说明，让有背景的读者快速回溯来源]
```

禁止把技术注记展开成独立段落或独立技术分析节。

## 写作要求

- `先看懂这件事` 必须短、准、直接，三条判断都应是读者能默念的短句
- 流转节必须先说"起点是什么、关键转折在哪里、终点是什么"
- 生命周期节必须先说"对象经历了哪几个关键阶段，每个阶段意味着什么"
- 模块节必须先说"谁承接、谁加工、谁输出"
- 每节都要有至少一句解释"为什么系统会这样组织"
- 散文骨架优先：每节先写 2-4 句散文，再决定是否补列表或表格
- 节标题必须是读者视角的导航判断句，不是操作备注

## 禁止项

- 不要直接基于 `conclusion-pack` 写报告，必须经过 `shaping-pack`
- 不要把规则清单作为正文核心
- 不要出现类名、方法名、SQL 语句作为正文主叙事
- 不要把代码调用链写成正文段落
- 不要把模块协作写成类/方法调用关系
- 不要把众所周知的步骤画进图里
- 不要让技术注记发展成独立的技术分析段落
- 不要为了模板完整保留空壳章节
- 不要把未确认内容写成确定结论
- 不要让子代理代写最终报告

## 一句话原则

**最终报告是主 agent 用 shaping-pack 写成的业务理解作品：先让技术人员看懂业务是什么，再让他们看懂业务怎么流、怎么演化、为什么这样组织，最后让他们能够回溯和验证。**
