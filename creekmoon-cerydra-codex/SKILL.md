# Cerydra Codex - 代码规则提取器

## 技能定位

这个技能只做一件事：**从现有代码中提炼某个维度的稳定约定，输出为一份可复用的规则文件。**

输出格式固定，文件命名固定（`中文名_rule.md`），内容始终偏向"约束表达"而不是"分析说明"。

---

## 调用前必须先确认

调用此技能前，必须先说清楚以下信息。信息不全时，先追问，不要直接开始归纳。

```
规则名称：    例如"承运商接入标准"、"通用CRUD规范"、"订单模块约定"
规则维度：    这份规则约束的是什么（见下方"规则类型"）
使用场景：    在哪些开发情况下需要遵守这份规则
适用范围：    涉及哪些目录、文件、模块
样本代码位置：至少给出大概目录、模块或入口文件；如果用户说不清代表文件，技能再自行选样本
保存位置：    规则文件准备保存到哪里；如果用户未指定，默认保存到 `.cursor/rules/`
```

**规则类型必须先判断：**

- `模块级规则`：针对某个具体模块的内部约定
  - 例如：订单模块、承运商接入模块、支付模块
  - 归纳重点：目录结构、内部职责切分、命名、边界
- `横切规则`：针对多个模块共用的实现套路
  - 例如：通用 CRUD、异常处理、DTO 转换、日志约定、SDK 集成
  - 归纳重点：必须跨多个模块样本验证，不能把某一个模块的局部写法误当通用约定

**如果用户说不清楚类型，默认先当模块级规则处理，归纳后再判断是否有更广泛适用性。**

## 执行流程

### Phase 0：确认信息

收到调用请求后，先确认以下信息是否齐全：

1. 规则名称和类型（模块级 / 横切）
2. 适用场景和范围
3. 样本代码位置在哪里
4. 规则文件保存位置在哪里

如有缺失，直接追问 1 到 2 个关键问题，拿到答案后再开始探索。

**追问优先级：**

1. 如果样本代码位置不清楚，优先问这个问题
2. 如果规则维度或使用场景模糊，再追问“这份规则到底约束什么、给谁用”
3. 保存位置若未指定，可直接采用默认值 `.cursor/rules/`，不必额外追问

---

### Phase 1：工具导向探索

目标是找到支撑规则条目的代码证据，不是做全面分析。

优先按以下工具顺序：

1. 用 `Glob` 找候选文件和目录
2. 用 `rg` 看命名模式、调用点、错误处理、测试、配置
3. 用 `Read` 精读少量代表文件

**采样策略：**

- 小模块或单一维度：读 `3-6` 个代表文件
- 中等模块或横切规则：读 `5-10` 个代表文件，且必须跨多个位置取样

**横切规则额外要求：**

必须跨至少 `2-3` 个不同模块取样，再归纳。不允许用单模块的写法代表全局约定。

**优先读这些文件：**

- 入口文件（路由、注册、配置）
- 主流程编排文件
- 核心能力文件（转换、校验、调用）
- 共享契约、类型定义、常量文件
- 至少一个测试文件

**如果样本不足（有效文件少于 3 个）：**

- 不输出强规则
- 在规则文件中标明"草案规则，待样本丰富后补齐"

---

### Phase 2：规则溯源（Evidence Map）

在写规则之前，先整理证据，不要一边读文件一边直接写规则条目。

`Evidence Map` 是**内部工作草稿**，用于判断哪些结论能写成强约束。它默认不出现在最终规则文件中；最终规则文件只保留必要的“参考实现”“依据说明”或“证据来源”。

每条待写规则都先回答：

| 规则点 | 证据文件 | 出现次数/范围 | 置信度 | 是否写成强约束 |
|--------|----------|--------------|--------|----------------|
| Parser 只做字段映射，不写流程逻辑 | `QuoteParser.java`, `BookingParser.java`, `TrackParser.java` | 3个文件一致 | 高 | 是 |

**置信度判断：**

- `高`：同一模式在多个关键文件里一致出现
- `中`：有明确样本，但覆盖面有限
- `低`：只在单个文件或局部片段出现

**只有置信度"高"的规则才能写成"必须/禁止"。中置信度的写"建议"，低置信度的不输出或标为草案。**

---

### Phase 3：生成规则文件

完成 Phase 2 后，按下面的模板生成规则文件，保存为 `{中文规则名}_rule.md`。

**保存位置规则：**

1. 用户已指定目录时，保存到用户指定位置
2. 用户未指定时，默认保存到 `.cursor/rules/`
3. 如果仓库语境明显不是 Cursor 规则目录，再补问是否改存到其他文档目录

文件结构模板（按需裁剪，不要求所有章节都出现）：

```markdown
---
description: {一句话说明这份规则在什么场景下被遵守}
globs:
  - {适用文件路径，例如 src/main/java/com/example/order/**/*}
alwaysApply: false
---

# {中文规则名}

本规则描述 {维度} 在 {使用场景} 下的实现约定，以现有代码的稳定模式为准。

一、适用范围

二、目录/结构约定（模块级规则常用；横切规则可省略）

三、职责边界（适用于存在清晰角色分工的场景）

四、强约束

五、配置/命名/契约约定（按需保留，非必选）

六、例外情况

七、参考实现 / 证据来源
```

至少保留这些核心章节：

- `一、适用范围`
- `四、强约束`
- `六、例外情况`（如确无例外，可明确写“暂无”）
- `七、参考实现 / 证据来源`

如果某一章对当前规则维度不成立，直接省略，不要为了凑模板硬写。

## 规则文件写法要求

### 语气要求

规则正文必须偏"约束表达"，不要写成说明文、分析文或流水账。

多用以下关键词，少用大段段落：

| 关键词 | 适用场景 |
|--------|----------|
| `必须` | 强约束，违反会导致问题 |
| `禁止` | 明确禁区，代码中有反例或明显风险 |
| `建议` | 有依据但非强制 |
| `允许` | 例外情况的显式声明 |
| `位置：` | 文件或目录约定 |
| `命名：` | 命名模式 |
| `职责：` | 某个角色能做什么、不能做什么 |

### 结构要求

- 每一章节标题用中文数字（一、二、三…）
- 每条约束独立一行，不堆在段落里
- 强约束条目必须附说明（为什么这么要求，或违反后有什么后果）
- 如果某条规则依赖现有样本，可注明"参考 `{文件路径}`"

### 例外情况要求

不要忽略例外。例外情况越明确，规则的可信度越高。

---

## 规则类型差异

### 模块级规则写法重点

- 重点描述该模块内部的目录约定、职责边界、命名、配置
- 规则的适用范围限于该模块，不要泛化成全局
- 如果模块内部存在新旧两套风格，分开写，说明哪套是当前推荐
- 通常会保留“目录/结构约定”“职责边界”章节

### 横切规则写法重点

- 必须说明这条规则在哪些模块中已验证（列出具体参考实现）
- 规则条目应抽象到"与具体模块无关"的表达
- 如果不同模块存在实质性差异，不要强行合并；改为"建议"或分场景写
- 可以省略“目录/结构约定”，把重点放在“强约束”“契约约定”“例外情况”

---

## 特殊场景处理

### 新旧风格并存

不要强行统一。分开描述，给出当前推荐方向和依据。

### 框架强约束

区分"框架规定"和"团队自己选择"，重点写后者。

### 跨模块链路规则

如果规则本质上是一条跨模块调用链路，按"链路阶段"组织，每个阶段分别说明约定，不要按目录硬拆。

---

## 禁止项

- 不要把行业最佳实践冒充成"项目当前实践"
- 不要因为看到 `1` 个例子就写成"必须"约束
- 不要把模块局部写法写成全局通用规则
- 不要省略置信度低的结论（要么不写，要么标明"草案"）
- 不要默认全仓扫描，尤其在 monorepo 里

---

## 输出前检查清单

1. 规则名称和适用场景写清楚了
2. 每条"必须/禁止"都有明确证据支撑（置信度高）
3. "建议"类条目与"强约束"条目区分清楚
4. 模块级规则没有被错误泛化成全局约定
5. 横切规则有跨模块样本验证
6. 例外情况已明确写出
7. 文件命名格式为 `中文名_rule.md`
