# Cerydra Codex - 代码规则提取器

## 技能定位

这个技能只做一件事：**从现有代码中提炼某个维度的稳定约定，输出为一份可复用的规则文件，帮助后续"解决同一类问题"时有据可依。**

输出格式固定，文件命名固定（`中文名_rule.md`），内容偏向"约定表达"而不是"分析说明"——但**每一条约定都带强度分级**，不是清一色的"必须/禁止"。

这个技能的常见场景是：手头已有写好的代码，想沉淀出一份后续可复用的约定。**正因为样本往往很少，规则更要克制，不能把"现状"当成"最佳实践"写死。**

---

## 核心原则（贯穿全流程）

下面三条优先级最高，与后续任何步骤冲突时以这三条为准。

### 原则一：强度分级，不要一刀切"必须/禁止"

规则的杀伤力在于语气。"必须/禁止"会直接写死扩展性——一旦未来出现合理的新情况，这条规则就变成了阻碍。

因此每条约定都要落到一个强度档位（见下方"约定强度阶梯"），**样本越少、必要性越弱，语气越要往下降。** 宁可写"可参考"，也不要轻易写"必须"。

### 原则二：业务语义优先，先理解"想干什么"再下结论

不要照着现有目录/命名无脑固化。下结论前，先从**文件名、包名、类名、变量名**出发，推断这段代码"想表达什么业务意图"，再判断**当前的归类、命名、位置是否和这个意图一致**。

只有"业务语义上确实合理"的模式，才有资格上升为强约束。

> 示例：`utils` 包下目前只有一个 `SysAdminConverter.java`。
> - 表面归纳：放进 `utils` 包 → "推荐工具类放 utils"。
> - 业务语义校验：`Converter` 暗示这是**业务兼容/模板代码**，未来很可能扩展出多个 Converter；严格说它不是通用工具，更像 `converter` 包的成员。
> - 结论：当前归类**存疑**，不能写成"推荐"。最多写"可参考：现状放在 `utils`；从命名看更像 `converter` 包，未来扩展时可考虑迁移"。

### 原则三：现状可疑时，写出更优方向，而不是固化现状

当一条约定只能落到"可参考"档（样本稀少 / 业务语义存疑），且你从命名或业务语义看出了更合理的安排，**必须把它写出来**（"现状是 X；从命名看更像 Y；未来可考虑 Z"），不要假装现状就是答案。

## 调用前必须先确认

调用此技能前，必须先说清楚以下信息。信息不全时，先追问，不要直接开始归纳。

```
规则名称：    例如"承运商接入标准"、"通用CRUD规范"、"订单模块约定"
规则维度：    这份规则约束的是什么（见下方"规则类型"）
使用场景：    在哪些开发情况下需要遵守这份规则
适用范围：    涉及哪些目录、文件、模块
样本代码位置：至少给出大概目录、模块或入口文件；如果用户说不清代表文件，技能再自行选样本
保存位置：    规则文件准备保存到哪里；如果用户未指定，默认保存到 `.cursor/rules/`
```

**规则类型必须先判断：**

- `模块级规则`：针对某个具体模块的内部约定
  - 例如：订单模块、承运商接入模块、支付模块
  - 归纳重点：目录结构、内部职责切分、命名、边界
- `横切规则`：针对多个模块共用的实现套路
  - 例如：通用 CRUD、异常处理、DTO 转换、日志约定、SDK 集成
  - 归纳重点：必须跨多个模块样本验证，不能把某一个模块的局部写法误当通用约定

**如果用户说不清楚类型，默认先当模块级规则处理，归纳后再判断是否有更广泛适用性。**

## 执行流程

### Phase 0：确认信息

收到调用请求后，先确认以下信息是否齐全：

1. 规则名称和类型（模块级 / 横切）
2. 适用场景和范围
3. 样本代码位置在哪里
4. 规则文件保存位置在哪里

如有缺失，直接追问 1 到 2 个关键问题，拿到答案后再开始探索。

**追问优先级：**

1. 如果样本代码位置不清楚，优先问这个问题
2. 如果规则维度或使用场景模糊，再追问“这份规则到底约束什么、给谁用”
3. 保存位置若未指定，可直接采用默认值 `.cursor/rules/`，不必额外追问

---

### Phase 1：工具导向探索

目标是找到支撑规则条目的代码证据，不是做全面分析。

优先按以下工具顺序：

1. 用 `Glob` 找候选文件和目录
2. 用 `rg` 看命名模式、调用点、错误处理、测试、配置
3. 用 `Read` 精读少量代表文件

**采样策略：**

- 小模块或单一维度：读 `3-6` 个代表文件
- 中等模块或横切规则：读 `5-10` 个代表文件，且必须跨多个位置取样

**横切规则额外要求：**

必须跨至少 `2-3` 个不同模块取样，再归纳。不允许用单模块的写法代表全局约定。

**优先读这些文件：**

- 入口文件（路由、注册、配置）
- 主流程编排文件
- 核心能力文件（转换、校验、调用）
- 共享契约、类型定义、常量文件
- 至少一个测试文件

**如果样本不足（有效文件少于 3 个）：**

- 不输出强规则
- 整份规则相关条目优先落在"推荐 / 可参考"档
- 在规则文件中标明"草案规则，待样本丰富后补齐"

---

### Phase 1.5：业务语义校验（下结论前的必做步骤）

读完样本、写证据图之前，对关键命名做一次业务语义校验。这是把"照搬现状"和"提炼合理约定"区分开的关键步骤。

对每个准备写进规则的目录 / 包 / 类 / 命名，问自己三个问题：

1. **它想干什么？** 从文件名、包名、类名、变量名推断业务意图。
2. **现状和意图一致吗？** 当前的归类、命名、位置，是否真的匹配这个意图？还是只是"暂时这么放着"？
3. **会不会限制扩展？** 如果未来同类东西变多，现在这个安排还成立吗？

校验结果落到一个标签，直接影响后续强度：

- `契合`：命名 / 职责 / 位置与业务语义一致，扩展方向清晰 → 有资格写强约束
- `存疑`：命名暗示的职责与当前归类不一致，或当前写法可能限制扩展 → 最多写"可参考"，并记下更合理的安排
- `待判断`：信息不足，无法确认 → 不写强约束

> 复用上面的例子：`utils` 包下只有 `SysAdminConverter` → "Converter 是业务兼容代码、未来会变多" → 当前归类**存疑** → 相关条目最多"可参考"，并注明"未来可考虑归入 `converter` 包"。

---

### Phase 2：规则溯源（Evidence Map，双轴判断）

在写规则之前，先整理证据，不要一边读文件一边直接写规则条目。

`Evidence Map` 是**内部工作草稿**，用于判断每条结论该落到哪个强度档。它默认不出现在最终规则文件中；最终规则文件只保留必要的"参考实现""依据说明"或"证据来源"。

强度不由单一维度决定，而是**两个轴一起看**：

- **证据强度**：这个模式有多少样本支撑、是否一致（来自 Phase 1）
- **业务契合度**：这个模式在业务语义上站不站得住（来自 Phase 1.5）

每条待写规则都先填这张表：

| 规则点 | 证据文件 | 证据强度 | 业务契合度 | 落到哪一档 |
|--------|----------|----------|------------|------------|
| Parser 只做字段映射，不写流程逻辑 | `QuoteParser.java`, `BookingParser.java`, `TrackParser.java` | 强（3个一致） | 契合 | 必须 |
| Converter 放在 utils 包 | `SysAdminConverter.java` | 弱（仅1个） | 存疑 | 可参考（并注明应归入 converter） |

**证据强度判断：**

- `强`：同一模式在多个关键文件里一致出现
- `中`：有明确样本，但覆盖面有限
- `弱`：只在单个文件或局部片段出现

**两轴 → 强度档位映射（核心规则）：**

| 证据强度 \ 业务契合度 | 契合 | 存疑 | 待判断 |
|----------------------|------|------|--------|
| 强 | **必须 / 禁止** | 推荐（写明存疑点） | 推荐 |
| 中 | 推荐 | 可参考 | 可参考 |
| 弱 | 推荐 / 可参考 | **可参考**（必须附更优方向） | 可参考 / 不写 |

**铁律：**

- 只有"证据强 + 业务契合"才允许写"必须 / 禁止"。
- 一旦业务契合度判为"存疑"，无论样本多少，都不写"必须 / 禁止"，最多"推荐"并写明存疑点。
- "证据弱 + 存疑"只能写"可参考"，且必须附上更合理的方向。

---

### Phase 3：生成规则文件

完成 Phase 2 后，按下面的模板生成规则文件，保存为 `{中文规则名}_rule.md`。

**保存位置规则：**

1. 用户已指定目录时，保存到用户指定位置
2. 用户未指定时，默认保存到 `.cursor/rules/`
3. 如果仓库语境明显不是 Cursor 规则目录，再补问是否改存到其他文档目录

文件结构模板（按需裁剪，不要求所有章节都出现）：

```markdown
---
description: {一句话说明这份规则在什么场景下被遵守}
globs:
  - {适用文件路径，例如 src/main/java/com/example/order/**/*}
alwaysApply: false
---

# {中文规则名}

本规则描述 {维度} 在 {使用场景} 下的实现约定，以现有代码的稳定模式为准。
每条约定带强度标注：[必须]/[禁止] 为强约束，[推荐] 为默认做法，[可参考] 仅为现状参考、不构成背书。

一、适用范围

二、目录/结构约定（模块级规则常用；横切规则可省略）

三、职责边界（适用于存在清晰角色分工的场景）

四、约定条目（每条前面标注强度：[必须]/[禁止]/[推荐]/[可参考]）

五、配置/命名/契约约定（按需保留，非必选）

六、例外情况

七、待确认 / 可演进点（样本少或业务语义存疑的条目，连同更优方向写在这里）

八、参考实现 / 证据来源
```

约定条目示例（注意强度标注与"可参考"档写法）：

```markdown
四、约定条目

[必须] Parser 只做字段映射，不写业务流程逻辑。
  依据：QuoteParser / BookingParser / TrackParser 三处一致。违反会让流程逻辑散落，难以维护。

[推荐] 新增转换类命名以 Converter 结尾。
  依据：现有转换类均如此命名，语义清晰。

[可参考] 转换类目前放在 utils 包（参考 SysAdminConverter.java）。
  说明：从命名看 Converter 更像业务兼容代码、未来可能扩展出多个，严格说不属于通用工具类。
  更优方向：未来转换类增多时，可考虑独立的 converter 包，而非统一塞进 utils。
```

至少保留这些核心章节：

- `一、适用范围`
- `四、约定条目`
- `六、例外情况`（如确无例外，可明确写"暂无"）
- `八、参考实现 / 证据来源`

如果某一章对当前规则维度不成立，直接省略，不要为了凑模板硬写。
"可参考"档的条目若带有更优方向，归到 `七、待确认 / 可演进点`，避免和强约束混在一起误导读者。

## 规则文件写法要求

### 语气要求

规则正文偏"约定表达"，不要写成说明文、分析文或流水账。但**约定 ≠ 强制**，要按强度阶梯选词，不要清一色"必须/禁止"。

**约定强度阶梯（从强到弱）：**

| 强度 | 关键词 | 含义 | 触发条件 |
|------|--------|------|----------|
| 强约束 | `必须` / `禁止` | 必须遵守，违反会出问题 | 证据强 + 业务契合（两者缺一不可） |
| 推荐 | `推荐` / `建议` | 默认这么做，但允许有理由地偏离 | 有样本支撑 + 业务语义站得住，但样本面或必要性不足以强制 |
| 可参考 | `可参考` / `可借鉴` | 仅作为现状参考，**不构成背书** | 样本稀少（≈1个）/ 或当前写法业务语义存疑 / 无法判断是否最佳 |
| 例外 | `允许` | 例外情况的显式声明 | — |

**选词纪律：**

- 默认往低档写。拿不准就降一档，"可参考"永远比错误的"必须"安全。
- `推荐` 与 `可参考` 的区别：`推荐`是"这么做更好、值得跟"；`可参考`是"现状如此、给你个参照，但不保证它合理"。
- 用"可参考"时，若看出更合理的安排，必须一并写出（见原则三）。

**其它结构关键词（与强度无关，用于标注约定类型）：**

| 关键词 | 适用场景 |
|--------|----------|
| `位置：` | 文件或目录约定 |
| `命名：` | 命名模式 |
| `职责：` | 某个角色能做什么、不能做什么 |

### 结构要求

- 每一章节标题用中文数字（一、二、三…）
- 每条约定独立一行，不堆在段落里
- 每条约定前标注强度：`[必须]` / `[禁止]` / `[推荐]` / `[可参考]`
- 强约束（必须/禁止）条目必须附说明（为什么这么要求，或违反后有什么后果）
- 可参考条目若有更优方向，必须写出来，不要只留一句"现状如此"
- 如果某条规则依赖现有样本，可注明"参考 `{文件路径}`"

### 例外情况要求

不要忽略例外。例外情况越明确，规则的可信度越高。

## 规则类型差异

### 模块级规则写法重点

- 重点描述该模块内部的目录约定、职责边界、命名、配置
- 规则的适用范围限于该模块，不要泛化成全局
- 如果模块内部存在新旧两套风格，分开写，说明哪套是当前推荐
- 通常会保留“目录/结构约定”“职责边界”章节

### 横切规则写法重点

- 必须说明这条规则在哪些模块中已验证（列出具体参考实现）
- 规则条目应抽象到"与具体模块无关"的表达
- 如果不同模块存在实质性差异，不要强行合并；降级为"推荐 / 可参考"或分场景写
- 可以省略"目录/结构约定"，把重点放在"约定条目""契约约定""例外情况"

---

## 特殊场景处理

### 新旧风格并存

不要强行统一。分开描述，给出当前推荐方向和依据。

### 框架强约束

区分"框架规定"和"团队自己选择"，重点写后者。

### 跨模块链路规则

如果规则本质上是一条跨模块调用链路，按"链路阶段"组织，每个阶段分别说明约定，不要按目录硬拆。

---

## 禁止项

- 不要把行业最佳实践冒充成"项目当前实践"
- 不要因为看到 `1` 个例子就写成"必须"约束（最多"可参考"）
- 不要用绝对语气把扩展性写死——拿不准就降档，别赌未来不会变
- 不要跳过业务语义校验，照搬现有目录 / 命名就直接下结论
- 不要把业务语义存疑的现状写成"推荐"（应降到"可参考"并写出更优方向）
- 不要把模块局部写法写成全局通用规则
- 不要省略证据弱的结论（要么不写，要么标明"草案 / 可参考"）
- 不要默认全仓扫描，尤其在 monorepo 里

---

## 输出前检查清单

1. 规则名称和适用场景写清楚了
2. 每条约定都标注了强度（必须 / 禁止 / 推荐 / 可参考）
3. 每条"必须/禁止"都满足"证据强 + 业务契合"，且附了说明
4. 做过业务语义校验：命名 / 包名 / 类名暗示的意图与归类一致；存疑处已降档
5. "可参考"档条目，凡看出更优方向的都写出来了
6. 模块级规则没有被错误泛化成全局约定
7. 横切规则有跨模块样本验证
8. 例外情况已明确写出
9. 文件命名格式为 `中文名_rule.md`
