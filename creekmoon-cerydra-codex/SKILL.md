# Cerydra Codex - 代码范式提取器

## 核心原则

**禁止预设架构**。每个项目、每个模块都有自己的组织方式。本技能的核心是：
1. **探索** - 看代码实际怎么写的
2. **归纳** - 找出重复出现的模式
3. **抽象** - 提炼成可描述、可遵循的规范
4. **表达** - 用具体项目的例子说明通用原则

---

## Phase 1: 探索（Exploration）

### 1.1 定位模块边界

首先确定要分析的功能范围：

```
用户说"分析订单模块"→ 问清楚：
- 订单的入口在哪里？（路由/Controller/API端点）
- 涉及哪些子功能？（创建/查询/状态变更/导出）
- 核心文件大概在哪几个目录？
```

**探索命令（根据语言调整）：**
```bash
# 看目录结构（不预设层级）
find . -type d -name "*order*" -o -name "*Order*" | head -20

# 看文件分布
tree -L 3 src/ 或 ls -R src/

# 看入口点
ls *Controller* *Handler* *Router* *API* 2>/dev/null
```

### 1.2 识别组织模式

**不要问"有没有Service层"，而是问：**

1. **文件如何分组？**
   - 是按功能（order/payment/user）？
   - 是按类型（controllers/models/utils）？
   - 还是混合方式？

2. **代码单元（函数/类/模块）的职责如何划分？**
   - 有的文件只做HTTP处理
   - 有的只做数据库操作
   - 有的只做业务计算
   - 有的混合了多种职责
   - → 记录实际的职责边界，而不是套用"标准三层"

3. **数据如何流转？**
   ```
   HTTP请求 → ? → ? → 数据库
             ↓
          中间经过了哪些转换？
          每层做了什么处理？
   ```

4. **错误如何处理？**
   - 在哪一层捕获？
   - 如何传递？
   - 如何返回给调用方？

### 1.3 提取命名模式（从代码中归纳）

**不要预设命名规则，而是看实际代码：**

```bash
# 看类/文件命名模式
grep -r "^class\|^def\|^function\|^export" --include="*.py" --include="*.js" --include="*.ts" --include="*.java" | head -50

# 看方法命名模式
grep -r "def .*\|function .*\|public.*(" --include="*.py" --include="*.js" --include="*.ts" --include="*.java" | head -50
```

**记录观察到的模式：**
```
发现模式：
├── 查询类方法常以 get/query/find/list 开头
├── 写入类方法常以 create/insert/save/update/delete 开头
├── 批量操作常以 batch/bulk 开头
├── 异步操作常以 async 结尾或用 Promise
└── 私有方法常以 _ 开头（Python）或 private 修饰（Java）
```

### 1.4 分析依赖关系

**关键问题：**
1. 哪些代码单元被多处复用？（提取成通用）
2. 哪些代码只在特定场景使用？（内聚性）
3. 依赖是单向还是双向？
4. 有没有循环依赖？

## Phase 2: 抽象（Abstraction）

### 2.1 识别工程共性（跨语言）

基于探索结果，抽象出**不依赖特定语言/框架**的原则：

#### A. 模块化原则
```
观察到的现象 → 抽象原则
─────────────────────────────────────────────────
订单相关代码都在 order/ 目录下    → 内聚性：相关代码物理上靠近
订单创建用了单独的文件           → 单一职责：一个模块只做一件事
多个地方用到了同一个价格计算     → DRY：重复逻辑抽取共享
```

#### B. 接口契约（输入/输出约定）
```
观察到的现象 → 抽象原则
─────────────────────────────────────────────────
每个函数都检查参数是否为null    → 防御式编程：前置校验
错误信息都有统一的格式          → 一致性：错误契约标准化
异步函数都返回 Promise          → 可预测性：返回值类型一致
```

#### C. 职责分离（不是"三层"，而是"关注点分离"）
```
发现项目实际采用的方式：

方式A - 水平分层：
  controllers/   ← 处理HTTP
  services/      ← 处理业务逻辑
  repositories/  ← 处理数据
  → 适合：业务逻辑复杂，需要清晰分层

方式B - 垂直切片：
  order/
    ├── api.js
    ├── logic.js
    └── db.js
  payment/
    ├── api.js
    ├── logic.js
    └── db.js
  → 适合：功能相对独立，团队按功能划分

方式C - 混合模式：
  → 记录实际的组织逻辑，不强套标准
```

#### D. 变化点隔离
```
观察：哪些配置/逻辑经常变动？
├── 第三方API地址 → 提取到配置文件
├── 业务规则（如折扣计算）→ 提取到策略类/配置
└── 日志格式 → 提取到中间件/拦截器
```

## 1. 代码组织

### 1.1 目录结构
```
{用树形图展示实际结构}
```

**组织逻辑**：
解释为什么这么组织（例如：
"按业务功能垂直划分，每个功能目录自包含，
便于功能级别的代码隔离和团队协作"）

### 1.2 职责划分

| 角色 | 典型文件 | 职责 | 禁止 |
|-----|---------|-----|-----|
| API层 | *.controller.ts | 处理HTTP请求/响应转换 | 禁止直接操作数据库 |
| 业务层 | *.service.ts | 实现业务规则 | 禁止依赖HTTP对象 |
| 数据层 | *.repository.ts | 数据持久化 | 禁止包含业务逻辑 |

### 1.3 依赖规则
- ✅ API层可依赖业务层和数据层
- ✅ 业务层可依赖数据层，不可依赖API层
- ✅ 共享工具可被任何层依赖
- ❌ 禁止业务层之间直接依赖

## 2. 命名约定

### 2.1 观察到的模式
```
查询单条：get{Entity} / find{Entity}ById
查询多条：list{Entities} / query{Entities}
创建：create{Entity} / insert{Entity}
更新：update{Entity} / patch{Entity}
删除：delete{Entity} / remove{Entity}
批量：batch{Operation}{Entities}
```

### 2.2 语义区分
- `get` vs `find`：get 期望一定存在，find 允许不存在
- `update` vs `patch`：update 全量更新，patch 部分更新
- `delete` vs `remove`：delete 硬删除，remove 软删除

## 3. 接口契约

### 3.1 输入规范
- 参数校验在入口处完成（API层）
- 业务层假设输入已校验，只做业务规则校验

### 3.2 错误处理
```typescript
// 统一错误格式
{
  code: "ORDER_NOT_FOUND",
  message: "订单不存在",
  details: { orderId: "123" }
}
```

### 3.3 返回值规范
- 查询单条：返回对象或 null
- 查询多条：返回数组（空数组表示无结果）
- 创建：返回创建后的完整对象
- 更新/删除：返回影响行数或布尔值

## 4. 最佳实践

### 4.1 复用
- 通用工具放在 `shared/utils/`
- 业务相关复用放在模块内 `internal/`

### 4.2 配置
- 环境相关配置通过环境变量注入
- 业务常量定义在模块内 constants.ts

### 4.3 测试
- 测试文件与被测文件同目录，后缀 `.test.ts`
- 单元测试覆盖核心业务逻辑
- 集成测试覆盖API端点

### 3.2 生成原则

1. **基于事实，不臆造**
   - 每个规范点都要有代码依据

2. **具体且可执行**

3. **承认多样性**

4. **语言无关表达**
   - 不说"Controller 层"

## 特殊场景处理

### 场景A：发现多种风格并存

**处理方式**：
```markdown
## 风格说明

本项目存在两种实现风格：

### 风格A（旧代码）
- 特征：...
- 位置：src/legacy/...
- 状态：维护模式，新功能不采用

### 风格B（新代码）
- 特征：...
- 位置：src/modules/...
- 状态：推荐用于新功能

### 迁移计划（如有）
计划逐步将风格A迁移到风格B，优先级...
```

### 场景B：代码量太少难以归纳

**处理方式**：
- 基于现有代码 + 行业最佳实践
- 明确标注哪些是"当前实践"，哪些是"建议实践"
- 生成"草案"版本，待代码丰富后更新

### 场景C：框架强制结构

**处理方式**：
- 明确标注"此部分由框架强制，不可更改"
- 重点描述框架约束下的可配置点

## 使用流程

用户说："分析XX模块，生成规范"

1. **澄清范围**
   - "XX模块涉及哪些功能？"
   - "核心代码大概在哪个目录？"
   - "有没有现有的参考文档？"

2. **探索代码**（Phase 1）
   - 看目录结构
   - 看关键文件
   - 归纳模式

3. **抽象原则**（Phase 2）
   - 识别组织逻辑
   - 提取命名模式
   - 总结接口契约

4. **生成文档**（Phase 3）
   - 按规范框架生成
   - 包含具体代码示例
   - 标注不确定的地方

5. **确认输出**
   - 询问用户："这个规范是否符合实际？"
   - 根据反馈调整
