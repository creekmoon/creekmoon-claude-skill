# Herta 深度研究

> 只做一件事：不惜代价地深度研究。

这个 skill 不把“代码逻辑”“静态结构”“业务关系”割裂成三件事，而是把它们视为理解同一个业务场景的不同证据面。

核心目标只有一个：**把一个业务场景研究透，并把结论用读者能消化的方式交付出来。**

一旦进入这个 skill，就默认进入**高强度、高成本、证据优先**的研究模式，不再区分轻量版或标准版。

---

## 触发门禁

本 skill 是 `opt-in only`。

只有在以下情况之一出现时，才允许触发：

- 用户显式点名 `creekmoon-herta-deepresearch`
- 用户显式说“请用深度研究 skill”“请用这个 research skill”
- 用户通过 slash 命令、附加 skill、或其他明确方式指定要用这个 skill

反过来，即使问题看起来很像研究任务，也**不要自动触发**：

- 不要因为出现“分析”“梳理”“研究”这类词就自动启用
- 不要因为任务复杂就擅自启用
- 不要因为环境支持子代理就擅自启用

如果用户没有显式调用，本 skill 一律不使用。

---

## 适用定位

当且仅当触发门禁已满足，且用户真正想搞清楚的是以下问题时，使用本 skill：

- 这套系统为什么会这样跑
- 一个业务场景从输入到结果到底经历了什么
- 某段代码逻辑、某组静态结构、某种业务关系分别在这个场景里扮演什么角色
- 业务口径和技术实现是否一致
- 哪些规则是事实，哪些只是推断，哪些仍未确认

不要把研究对象误当成 skill 类型：

- 用户提到“代码逻辑”，不一定是在做纯代码分析，可能是在研究业务决策链
- 用户提到“表结构/配置”，不一定是在做纯数据设计，可能是在研究规则载体
- 用户提到“角色关系/上下游依赖”，不一定是在做组织分析，可能是在研究场景闭环

如果真实目标是**解释一个业务场景如何运转**，就应该使用本 skill。

### 何时不要自动用

以下情况即使“有点像研究”，也不要自动触发本 skill：

- 用户只是想看某个类、某个方法、某张表、某个字段的局部解释
- 用户只是在做简单问答，没有要求形成结构化研究结论
- 用户只是做一次性调试、修 bug、查报错
- 用户的问题局限在单文件或单函数，且不涉及业务场景闭环
- 用户只需要快速答案，不需要高成本取证和双视图输出

如果用户**已经显式调用**了本 skill，但问题仍然太碎、太局部、太浅，不要退回普通回答；先把题目提升为一个可研究的场景问题，再开始深挖。

## 先做分型

在写任何结论前，先判断这次研究的主类型。分型不是为了限制输出，而是为了决定你先看什么、重点写什么。

| 类型 | 核心问题 | 常见信号 | 优先关注 |
|------|----------|----------|----------|
| `logic` 逻辑型 | 它什么时候触发、怎么判断、怎么流转、失败会怎样 | “为什么这样跑”“规则怎么判”“分支在哪”“默认/回退是什么” | 触发条件、阶段流转、关键分支、默认值、容错、后果 |
| `structure` 结构型 | 这套东西由什么组成、字段/枚举/配置如何约束行为 | “字段是什么意思”“有哪些状态”“配置怎么影响行为”“结构怎么组织” | 对象层级、字段语义、必填/默认/枚举、来源、派生、约束 |
| `relation` 关系型 | 谁和谁发生关系、谁负责什么、上下游如何约束 | “谁依赖谁”“责任边界”“谁提供/谁消费”“哪些角色受影响” | 参与方、职责边界、信息流、状态流、依赖、手工介入点 |
| `mixed` 混合型 | 单靠一种视角解释不完整 | 同时问规则、结构、关系；或研究场景天然跨多层 | 先定主类型，再用次类型补足 |

判定规则：

- 用户主要在问“怎么运行、为什么这样判、结果会怎样”，主类型判为 `logic`
- 用户主要在问“由什么组成、字段/配置怎么约束”，主类型判为 `structure`
- 用户主要在问“谁和谁如何协作、谁承担结果”，主类型判为 `relation`
- 现实业务里 `mixed` 很常见，但不要平均用力；必须明确一个主类型

## 读者模式

这个 skill 支持三种输出模式。研究层只做一套，表达层按读者切换。

| 模式 | 适用对象 | 写法要求 |
|------|----------|----------|
| `business` | 业务、产品、运营、非技术管理者 | 用业务术语解释规则、后果、边界；不出现类名、方法名、SQL、接口路径 |
| `technical` | 技术负责人、架构师、核心开发 | 讲清实现入口、责任边界、关键链路、结构载体、配置影响、改动风险 |
| `dual` | 同时面向业务与技术 | 共享一份结论底稿，拆成“业务视图 + 技术视图”双层交付 |

默认规则：

- 未指定读者，默认 `dual`
- 用户明确只要业务结论，切 `business`
- 用户明确只要技术审阅材料，切 `technical`

重要约束：

- `business` 视图禁止泄漏实现细节
- `technical` 视图不能只堆代码名词，必须解释这些实现对业务意味着什么
- `dual` 模式下，业务视图和技术视图的结论必须一致，不能各说各话

---

## 研究姿态

本 skill 没有 `standard / heavy` 之分。

一旦触发，就按高强度深度研究执行：

- 优先完备性、证据密度和结论置信度，不优先速度
- 优先建立场景级闭环，不满足于局部解释
- 优先多面取证、交叉验证、反向否证，不满足于单点证据
- 如果环境支持子代理，默认采用“主代理统筹 + 并行子代理分面取证 + 主代理统一收口”的方式
- 如果环境不支持子代理，单代理也要按同样的研究标准顺序完成，不得降级成浅层概览

## 证据规则

### 1. 结论强度不能超过证据强度

所有关键结论都要归入以下三层之一：

| 结论层级 | 含义 | 允许写法 |
|----------|------|----------|
| `事实` | 已被代码行为、配置事实、文档说明、稳定结构直接证明 | 可以直接下结论 |
| `证据支持的推断` | 多个证据共同指向，但仍带解释边界 | 可以写，但必须说明依据和限制 |
| `未确认` | 证据不足、证据冲突、链路未覆盖 | 不能伪装成定论，必须说明缺什么证据 |

### 2. 以下内容只能算线索，不能单独当结论

- 命名看起来像某个业务概念
- 单个 DTO 字段或注释
- 单处枚举值
- 某个适配器/客户端/第三方模型名称
- 局部代码片段脱离上下文的表现

### 3. 对高风险结论更保守

以下类型的结论必须更谨慎：

- 系统业务边界
- 核心责任归属
- 是否属于统一口径 / 全局规则
- 是否允许绕过某个前置环节
- 某个字段或配置是否为唯一真相来源

如果没有足够证据，宁可写“未确认”，也不要补全想象中的闭环。

## 研究流程

### Phase 0：先确认触发门禁

先确认这次是用户显式调用，而不是模型擅自触发：

- 未显式调用：不要使用本 skill
- 已显式调用但题目太散：先把问题收敛成可研究的场景
- 已显式调用且题目清楚：直接进入高强度研究流程

### Phase 1：先把题目收敛成一个场景

在动手搜代码、搜结构、搜配置之前，先明确：

- 本次研究的**业务场景名称**
- 这次真正要回答的**主问题**
- 场景的起点和终点
- 主要参与方 / 主要输入 / 主要输出
- 本次不研究什么

如果题目太散，先把它改写成一句研究问题：

`在 [场景] 中，系统如何基于 [输入/关系/结构] 做出 [决策/流转/结果]，以及这一行为对 [业务/技术] 有什么影响？`

### Phase 2：确定主类型与次类型

输出一张小表，作为后续研究导航：

| 项目 | 结论 |
|------|------|
| 主类型 | `logic / structure / relation / mixed` |
| 次类型 | 若为 `mixed`，列出 1-2 个次类型 |
| 读者模式 | `business / technical / dual` |
| 本次重点 | 本次真正想研究透的 3-5 个问题 |

### Phase 3：收集证据，不急着写正文

先铺证据，再写结论。建议优先收集：

- 业务文档、说明、注释
- 关键入口、核心调用链、状态迁移
- 数据结构、枚举、配置项、映射关系
- 上下游依赖、外部系统边界、手工操作点
- 失败路径、回退路径、容错策略

如果是 `logic` 主导，优先找：

- 触发条件
- 阶段划分
- 分支判定
- 默认值 / 回退 / 拒绝条件
- 输出结果 / 副作用 / 失败后果

如果是 `structure` 主导，优先找：

- 结构层级
- 字段/枚举/配置的业务含义
- 必填/默认/派生/互斥/约束
- 哪个结构是真正驱动行为的

如果是 `relation` 主导，优先找：

- 参与方与职责
- 谁提供输入、谁做判断、谁承担结果
- 信息流 / 状态流 / 交付流
- 边界冲突、灰区、人工补偿点

本 skill 默认按“分面取证、统一收口”的方式执行：

- 主代理先定义研究问题、主类型、读者模式、交付目标
- 若环境支持子代理，默认并行拉起 2-4 个子代理，而不是一个子代理包办全部研究
- 子代理只负责某一面证据收集与候选结论，不直接输出最终报告
- 主代理负责合并、去重、降级结论、处理冲突，并决定最终写法

默认的子代理分工：

| 子代理 | 关注面 | 必须返回 |
|--------|--------|----------|
| `logic-tracer` | 入口、阶段、状态、分支、默认/回退、失败路径 | 阶段表、关键判断点、失败后果、证据锚点、未确认项 |
| `structure-mapper` | 实体、字段、枚举、配置、映射规则 | 结构项清单、约束、默认/派生关系、驱动行为、证据锚点 |
| `relation-analyst` | 参与方、上下游、责任边界、信息/状态流 | 关系矩阵、依赖关系、人工介入点、边界冲突、证据锚点 |
| `gap-auditor`（可选） | 反向审查 | 哪些结论证据不足、哪里有冲突、哪些结论应降级为推断或未确认 |

子代理使用约束：

- 不要按文件夹机械拆任务，要按研究问题拆
- 不要让子代理直接写最终业务视图或技术视图
- 不要把“命名像什么”当成直接事实返回
- 子代理结果只能算“候选证据包”，不是最终结论

若环境不支持子代理，就按同样分面在单代理内顺序完成

### Phase 4：先写“候选结论清单”

正文之前，先形成结论底稿：

| 候选结论 | 类型 | 证据锚点 | 当前判断 | 正文去向 |
|----------|------|----------|----------|----------|
| `{结论}` | 事实 / 推断 / 未确认 | `{证据}` | 可写 / 需降级 / 暂缓 | 摘要 / 业务视图 / 技术视图 / 未确认项 |

这样做的意义是：

- 防止一边写一边脑补
- 防止业务结论和技术结论互相打架
- 让“未确认项”成为正式输出的一部分，而不是遗漏

### Phase 5：再组织成对应读者的输出

只有在候选结论清单基本稳定后，才进入正式报告。

- 一句话主结论
- 主类型 / 次类型
- 关键风险或最大不确定点

## 2. 研究范围与证据边界
| 项目 | 内容 |
|------|------|
| 研究问题 | |
| 起点 / 终点 | |
| 主要参与方 | |
| 证据来源 | |
| 未覆盖范围 | |

## 3. 业务视图
### 3.1 场景目标与参与方
### 3.2 输入约束 / 准入条件
### 3.3 核心规则与关键分支
### 3.4 上下游影响与业务后果

## 各类型的重点写法

### A. `logic` 逻辑型

必须回答：

- 什么条件下进入这个场景
- 系统按什么顺序处理
- 关键判断点在哪里
- 默认 / 回退 / 拒绝条件是什么
- 成功结果、失败结果、异常传播各是什么

推荐表：

| 阶段 | 触发条件 | 输入 | 系统行为 | 输出 | 失败后果 |
|------|----------|------|----------|------|----------|

### B. `structure` 结构型

必须回答：

- 这套结构由什么组成
- 哪些字段 / 枚举 / 配置真正影响行为
- 默认值、必填、互斥、派生规则是什么
- 结构变化会影响哪些业务判断

推荐表：

| 结构项 | 业务含义 | 约束 | 默认/派生 | 影响的行为 |
|--------|----------|------|-----------|------------|

### C. `relation` 关系型

必须回答：

- 谁参与这个场景
- 谁提供输入，谁负责判断，谁消费结果
- 信息 / 状态 / 交付如何流转
- 哪些关系会带来责任边界、冲突或灰区

推荐表：

| 参与方/对象 | 角色 | 输入/产出 | 依赖关系 | 失效后的影响 |
|-------------|------|-----------|----------|--------------|

### D. `mixed` 混合型

写法要求：

- 明确一个主类型，正文按主类型展开
- 次类型只用于补主类型解释不完的部分
- 不要平均分配篇幅，避免写成“什么都沾一点但都不透”

---

## 表达约束

### 对业务视图

- 只讲业务规则、业务后果、业务边界
- 用业务术语，不用类名、方法名、SQL、接口路径
- 如果技术事实很重要，也先翻译成业务含义再写

### 对技术视图

- 允许出现实现入口、模块、配置、结构和证据锚点
- 但不要堆砌源码细节；要说明“这对场景意味着什么”
- 优先解释边界、依赖、状态、风险，不要沦为代码目录导览

### 对双视图

- 业务视图回答“系统对业务表现出来什么”
- 技术视图回答“系统内部为什么会表现成这样”
- 两边结论必须互相映照，而不是简单重复

---

## 禁止项

- 未满足显式调用条件就自动触发本 skill
- 没分型就直接开始写长报告
- 没确定读者模式就混写业务术语和实现细节
- 只罗列代码入口，不解释业务意义
- 只罗列字段/结构，不解释它们怎么影响场景
- 只画关系，不解释关系背后的责任和后果
- 把命名、注释、单个字段当成最终事实
- 把“未确认项”藏掉，假装已经研究清楚
- 把整个任务完整丢给某一个子代理，主代理自己不做收口
