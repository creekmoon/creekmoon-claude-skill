# 实现类 / 风险类决策标尺

> 服务于决策循环的 ② 步：当决策是「怎么做才对」（实现类）或「能不能安全动」（风险类）时，用这里的信号判断该偏激进还是偏保守。
> 这些是**定性倾向信号**，不是分数。不要把它们加总成小数。
> 通用规则：**任一维度落在"偏保守 / 高风险"，就把整体裁决向保守靠一档。**

## 维度1：代码健康（实现类）

### 偏激进（可以放手做）
| 信号 | 检测方式 |
|------|---------|
| 测试覆盖率 > 70% | 运行覆盖率报告 |
| CI 连续 10 次绿色 | 查看 CI 历史 |
| 近 14 天无 P1/P2 Bug | 查看 Bug 跟踪 |
| TODO < 5 个 | grep TODO/FIXME |

### 偏保守（先稳住）
| 信号 | 检测方式 |
|------|---------|
| 测试覆盖率 < 40% | 覆盖率报告 |
| CI 红色或不稳定 | CI 历史 |
| 近 7 天有 P1 Bug | Bug 跟踪 |
| 目标模块零测试 | 文件检查 |
| 代码里有 "hotfix"/"workaround" | grep |

## 维度2：变更范围（实现类 + 风险类）

| 层级 | 风险 |
|------|------|
| UI/展示层 | 低 |
| 工具/辅助函数 | 低 |
| 业务逻辑 | 中 |
| API 契约/DTO | **高** |
| 数据库 Schema | **高** |
| 认证/授权/支付 | **极高（命中安全红线）** |

## 维度3：时间约束（实现类）

| 场景 | 倾向 |
|------|------|
| MVP/原型阶段 | 偏激进 |
| 活跃开发期 | 均衡 |
| 功能冻结/RC | 偏保守 |
| 生产环境/热修 | 偏保守 |

## 维度4：依赖风险（风险类——影响面）

| 信号 | 风险 |
|------|------|
| 自包含模块，无外部消费者 | 低 |
| 1-2 个内部消费者 | 低 |
| 3-5 个消费者 | 中 |
| >5 个消费者或公共接口 | **高** |
| 接口已冻结/已废弃 | **极高** |

## 维度5：可逆性（风险类——能不能回头）

| 变更类型 | 可逆性 | 倾向 |
|---------|-------|------|
| 纯新增（新文件/新端点/新模块）| 易回滚，不破坏现有功能 | 偏激进 |
| 修改现有代码 | 一般可回滚 | 基准 |
| 删除代码 | 可能有未知消费者 | 偏保守 |
| 删主数据 / 破坏性迁移 / 对外发布 | **难以回滚** | **强保守 + 查升级条件** |

**判读**：新增 API 端点（不影响已有）= 偏激进；修改已有 API 响应 = 基准；给已有表加字段 = 基准；删表/删字段/改 Schema = 偏保守。

## 安全覆盖（所有模式都生效）

涉及以下内容的改动，**自动保守处理，不受激进模式影响**：
- 用户认证/授权
- 支付/账单
- 用户个人信息(PII)
- 数据库主数据删除

这些命中 SKILL.md 的安全红线；若同时信息不足以判断，进入 ④ 升级判断。
