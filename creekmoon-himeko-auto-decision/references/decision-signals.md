# 5维度决策信号目录

## 维度1：代码健康

### 激进信号 (+1)
| 信号 | 检测方式 |
|------|---------|
| 测试覆盖率 > 70% | 运行覆盖率报告 |
| CI连续10次绿色 | 查看CI历史 |
| 近14天无P1/P2 Bug | 查看Bug跟踪 |
| TODO < 5个 | grep TODO/FIXME |

### 保守信号 (-1)
| 信号 | 检测方式 |
|------|---------|
| 测试覆盖率 < 40% | 覆盖率报告 |
| CI红色或不稳定 | CI历史 |
| 近7天有P1 Bug | Bug跟踪 |
| 目标模块零测试 | 文件检查 |
| 代码里有"hotfix"/"workaround" | grep |

## 维度2：变更范围

| 层级 | 风险 |
|------|------|
| UI/展示层 | 低 |
| 工具/辅助函数 | 低 |
| 业务逻辑 | 中 |
| API契约/DTO | **高** |
| 数据库Schema | **高** |
| 认证/授权/支付 | **极高（安全覆盖）** |

## 维度3：时间约束

| 场景 | 倾向 |
|------|------|
| MVP/原型阶段 | 激进 |
| 活跃开发期 | 均衡 |
| 功能冻结/RC | 保守 |
| 生产环境/热修 | 保守 |

## 维度4：依赖风险

| 信号 | 风险 |
|------|------|
| 自包含模块，无外部消费者 | 低 |
| 1-2个内部消费者 | 低 |
| 3-5个消费者 | 中 |
| >5个消费者或公共接口 | **高** |
| 接口已冻结/已废弃 | **极高** |

## 维度5：变更类型

| 类型 | 得分 | 理由 |
|------|------|------|
| 纯新增（新文件/新端点/新模块）| +1 | 不会破坏现有功能 |
| 修改现有代码 | 0 | 基准 |
| 删除代码 | -1 | 可能有未知消费者 |

**规则**：新增API端点（不影响已有）= +1；修改已有API响应 = 0；添加字段到已有表 = 0

## 安全覆盖（所有模式都生效）

涉及以下内容的改动，**自动保守处理**：
- 用户认证/授权
- 支付/账单
- 用户个人信息(PII)
- 数据库主数据删除

这些不受激进模式影响。
